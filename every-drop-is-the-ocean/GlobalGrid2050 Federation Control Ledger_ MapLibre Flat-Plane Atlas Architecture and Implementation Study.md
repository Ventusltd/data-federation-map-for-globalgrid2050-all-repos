# GlobalGrid2050 Federation Control Ledger — Architecture & Implementation Study

*A build-ready synthesis under contract GLOBALGRID2050_2D_ATLAS_STUDY_CONTRACT.txt v2026-06-30 (owner Vikram Kumar / Ventus Ltd; executor ChatGPT; independent auditor Claude; parallel agent Gemini). This study reconciles the two proven Ventus patterns and current open-source best practice; it does not re-survey the source repositories from scratch.*

## TL;DR

- **Build it now on MapLibre GL JS, but drive it non-geographically by treating abstract topology coordinates as the lon/lat plane.** The REPD Grid Atlas v8 architecture transfers almost entirely, because the federation atlas is *less* constrained than the Earth atlas; the single hard caveat is that MapLibre is permanently locked to Web Mercator (EPSG:3857) and has no Leaflet-style `CRS.Simple`, so the build-time layout must precompute `x_hint`/`y_hint` inside a safe lon ∈ [-180,180], lat ∈ [-80,80] window and never reproject.
- **The current verified data (16 nodes, ~30–35 deduplicated edges) renders instantly with plain GeoJSON sources** and stays buildable-now in `live_sandbox/federation_control_ledger/`; the scale tiers (client clustering → PMTiles vector tiles → deck.gl WebGL) are deferred behind named numeric thresholds and are not needed until the federation grows by roughly two-to-three orders of magnitude.
- **Truth stays in Parquet/DuckDB and is read-only in the UI.** The frontend may reorder, pin, collapse, and zoom but can never make a red node green, and the two unresolved findings (the camelCase `scanId|nodeId` key vs. the contract’s `repo_id` snake_case key law, and the `governance_dependency` edges still pointing at `globalgrid2050-hompage`) must be shown honestly as grey/flagged states — not fixed in this study.

## Key Findings

1. **MapLibre GL JS has no identity/Simple CRS — it is hard-wired to Web Mercator.** MapLibre maintainers confirm in issue #168 that the library “basically only speaks Web Mercator,” and issue #1228 (“Non-Earth maps/projections (flat maps)?”) confirms that Leaflet’s Simple-CRS pattern “is not possible in MapBox.”  The MapLibre Native docs state Web Mercator (EPSG:3857) is “its only supported projection.”  This is the single most important architectural constraint in the whole study.
1. **The documented community workaround makes a non-geographic atlas fully viable.** MapLibre contributor Kyle Barron’s guidance in issue #168 is to use “the lat/lon boundaries of `[-180, -90, 180, 90]` as an arbitrary canvas… convert your data… to their relative points in this canvas *without reprojecting*.”  Game maps, floor plans, blueprints and whiteboards already do this (issues #163, #6081). The only cost is Mercator’s latitudinal stretch, mitigated by confining nodes to roughly lat ∈ [-80, 80].
1. **MapLibre’s `CustomLayerInterface` (type:“custom”, renderingMode:“2d”) can draw arbitrary WebGL** — the camera matrix “projects spherical mercator coordinates to gl coordinates,”  so federation edges can be drawn as a custom 2D GL layer if the native `line` layer ever proves insufficient.
1. **The v8 declarative one-engine/many-filtered-views model transfers wholesale.** MapLibre’s expression system supports exactly the v8 idioms: `["interpolate",["linear"],["zoom"],…]` for zoom-scaled radius, `["interpolate",…,["get","importance_score"]]` for data-scaled radius, `["match",["get","status"],…]` for status colours, and `filter` expressions to slice one shared source into many layers. This is the heart of the “add one config object, never touch the engine” pattern.
1. **The current 16-node / ~30-edge dataset is trivially small for every renderer.** Per Cylynx’s “A Comparison of Javascript Graph / Network Visualisation Libraries,” using a 2015 MacBook: “SVG performance tops gives workable performance until it reaches 2k nodes and 2k edges. Canvas performance reaches the limit at 5k nodes and 5k edges while WebGL is usable until 10k nodes and 11k edges.” deck.gl’s official Performance guide reports that on 2015 MacBook Pros “most basic layers (like ScatterplotLayer) render fluidly at 60 FPS… up to about 1M (one million) data items, with framerates dropping into low double digits (10-20FPS) when the data sets approach 10M items” (and a hard ceiling where layers “crash during GPU buffer generation somewhere between 10M and 100M items”). The federation will not approach these limits for a long time, so the buildable-now sandbox needs nothing exotic.
1. **PMTiles is the correct deferred scale tier** — a single-file vector-tile archive served by HTTP range requests “free of a custom tile backend,” generated by tippecanoe and hosted on GitHub Pages. It preserves the contract’s “no backend at view time” law at the million-node scale.
1. **Sugiyama layered-DAG layout (via dagre, d3-dag, or ELK.js) is the right build-time `x_hint`/`y_hint` generator** for the dependency dimension, and it must be kept strictly separate from the single-parent containment hierarchy (`parent_federation`).
1. **The DuckDB → Parquet → JSON/GeoJSON export is a one-command `COPY` step in CI**, and the GitHub Actions cron-build → GitHub Pages deploy pattern is exactly the v8 pattern.

## Details

### Q1 — Does the v8 architecture transfer to a non-geographic atlas? (Recommendation + reasoning)

**Recommendation: YES — keep MapLibre GL JS as the engine, but operate it as a flat plane by writing abstract coordinates straight into the lon/lat slots.** Reasoning:

- The federation atlas is, as the contract observes, *easier* than the Earth atlas in one decisive respect: there is no fixed real-world geometry to honour. Node positions are assigned by us at build time, so we are free to choose coordinates that live happily inside the Web Mercator window.
- The blocking fact is that MapLibre cannot be given a custom/identity CRS today (Findings 1–2; MapLibre issues #168, #1228; MapLibre Native “Coordinate System” / “Architectural Problems” docs). So we do **not** fight Mercator — we adopt the documented “abstract canvas” workaround (Kyle Barron, issue #168): the build step normalises every `x_hint`/`y_hint` into lon ∈ [-180, 180], lat ∈ [-80, 80] and the engine treats them as ordinary points. No basemap tiles are loaded (the style uses a plain black background layer), so there is no “Russia in the upper-right” problem.
- **Why not a dedicated graph renderer (sigma.js / Cytoscape) or deck.gl as the primary engine?** Because the contract’s entire value proposition is *reusing the proven v8 engine and its config grammar*. MapLibre gives us, for free: `filter`-sliced shared sources, zoom-gated `minzoom`, data-driven `interpolate`/`match` expressions, popups-on-selection, lazy source loading, vector-tile/PMTiles scaling, and the exact same deploy story. A graph library would force a rewrite of all of that. deck.gl’s `OrthographicView` is genuinely non-geospatial and is the cleaner *math* (no Mercator distortion at all — it “creates a top-down view of the XY plane… usually used for rendering 2D charts in non-geospatial use cases”), so it is the recommended **escape hatch** for the WebGL scale tier — but it is not the buildable-now engine because it does not share the v8 config DNA.
- **Edges:** draw them as MapLibre `line` layers from a GeoJSON `LineString` source (one feature per edge, precomputed bend points). If global collision-free custom routing is ever needed, MapLibre’s `CustomLayerInterface` 2D mode (Finding 3) can draw edge geometry directly in WebGL.

**Verdict:** MapLibre-as-flat-plane is the right primary engine; deck.gl `OrthographicView` is the named escape hatch at scale; 3D is explicitly excluded.

### Q2 — Exact parts of REPD Grid Atlas v8 to copy

Copy verbatim (port, do not redesign):

- **The deep-frozen declarative config array.** One immutable array of layer GROUPS; each layer is a plain object. To add a dataset you add one object; the engine never changes.
- **One engine / many filtered views.** All node layers read ONE shared `nodes.json` (or its tiled equivalent); all edge layers read ONE shared `edges.json`; layers differ only by `node_filter`/`edge_filter` expressions on `repo_type`, `status`, `edge_type`, `federation_level`.
- **Layer groups + toggles with `[OK]`/`[WAIT]`/`[LOAD]` state tags**, coloured per layer, in a mobile-first dark control panel.
- **Lazy loading (`preload:false`)** — a layer fetches its slice only on first toggle. (MapLibre large-data guidance: split sources into chunks; set source `maxzoom` low; set layer `minzoom` > 0.)
- **`minzoom` gates** — fine-grained layers (files, evidence, commits) do not draw until you zoom in, exactly like v8’s 11kV-past-zoom-13.5 rule.
- **Status-derived colours from fixed maps** via `["match",["get","status"], "operational","#2ecc40", …]` — deterministic, never user-set. The four-colour legend (green Operational, yellow Under Construction, orange Consented, purple Applied) is reused as the federation status legend (with grey = unknown added as a legal state).
- **Data-derived size via `interpolate` on a data field** — v8 scales radius by capacity; the ledger scales radius/card-size by `importance_score`.
- **Popups only on selection** (e.g. “Repo | data | active | UK”).
- **Left-rail tool buttons**, adapted to topology: EXPORT CSV (export the current filtered node/edge set), RADIUS SEARCH → “neighbourhood search” (N hops from a node), POLY ZONE → “sector lasso,” STATUS COLOURS toggle, MEASURE → “dependency path length / hop count.”
- **The VENTUS header**: live SYSTEM TIME clock top-left, “2050 TARGET — 8585 DAYS” countdown top-right, four-colour status legend, wide-letter-spaced uppercase monospace/system wordmark exactly matching “VENTUS / CABLES & CONNECTIVITY®.” Public name “GlobalGrid2050 Federation Control Ledger”; strapline “Ventus Global Grid 2050 repository federation for an electrified future.” The internal codename appears in NO public file.
- **Static GitHub Pages deploy + GitHub Actions build/export job, no server at view time.** The standard pattern is a workflow with `on: { push, schedule: cron, workflow_dispatch }` that runs the DuckDB export then `actions/upload-pages-artifact` + `actions/deploy-pages` (or `peaceiris/actions-gh-pages`).

### Q3 — Exact parts of the World Cup app to copy + how to render connectors

Copy the *feel*, not the football:

- Dark full-screen mobile-first view; smooth horizontal pan; large readable cards; clean status chips; fullscreen + reset controls; connector paths between cards; tabbed stage/layer navigation; strong touch usability.
- Map World Cup ROUNDS→federation levels (tabs), TEAMS→nodes (cards), MATCHES→edges (connectors) — but with the binary-parent rule removed (see Q4).

**Connector rendering decision.** The graph-drawing literature gives a clean rule: **use orthogonal H/V/H “elbow” connectors for the containment/hierarchy edges and the directed dependency edges, and smooth cubic Béziers for dense or cross-cluster edges.** Orthogonal routing yields “good angular resolution and short links”  (arXiv 1204.4997, “Optimal Orthogonal Graph Drawing with Convex Bend Costs”) and “highly readable, grid-like diagrams”  (yWorks), and an eye-tracking study (Comparing the Readability of Graph Layouts) found orthogonal layouts impose “less link tracing effort.”  D3’s own `d3.link`/`linkHorizontal` generates “a smooth cubic Bézier curve… [whose] tangents… are either vertical or horizontal,” explicitly “suitable for visualizing links in a tree diagram.” Concretely: in MapLibre, precompute each edge’s geometry at build time — an orthogonal edge is a 4-point `LineString` (source → mid-x → mid-x → target), a Bézier edge is a densified polyline approximating the curve — and render via a `line` layer styled by `edge_type`. Bends are minimised because “bends obfuscate the readability of orthogonal drawings” (arXiv 1204.4997).

### Q5 — The full DATA CONTRACT (JSON schemas + Parquet mapping + manifest)

All files are static JSON (or PMTiles at scale) served from GitHub Pages. camelCase Parquet columns are mapped to snake_case JSON keys at export time by the DuckDB query (a `SELECT … AS …` rename), **without** altering the underlying scanner schema (out of scope).

**`nodes.json`** (FeatureCollection; geometry = `[lon,lat]` from `x_hint`,`y_hint`):

```
id            ← nodeId            (string; see key-law caveat)
label         ← repoName
node_type     ← nodeKind          (github_repo | external_source_or_service)
repo_type     ← repoType          (data|source_archive|homepage|ui|external|unknown)
federation_level ← derived        (0=domain … 4=file; build-time)
parent_federation ← derived       (single parent id; containment only)
cluster       ← derived           (domain/cluster id)
importance_score ← computed       (see Q9)
status        ← status            (active|referenced → mapped to legend states; unknown=grey)
status_reason ← derived           (human-readable why)
x_hint, y_hint ← layout           (abstract coords; normalised to lon/lat window)
min_zoom      ← derived           (zoom gate, Q6)
evidence      ← evidencePath/Text (provenance)
source_url    ← htmlUrl
swhid         ← optional          (Software Heritage SWHID; see below)
```

Also carried through for audit: `scanId`, `generatedUTC`, `methodVersion`, `pushedAt`, `updatedAt`, `sizeKb`, `visibility`, `archived`, `canonicalFilesPresent`, `defaultBranch`.

*On `swhid`:* the SoftWare Hash IDentifier was officially published as the ISO/IEC international standard **ISO/IEC 18670:2025 — “Information technology — SoftWare Hash IDentifier (SWHID) Specification V1.2”** on 23 April 2025 (Software Heritage). SWHIDs are intrinsic, content-based identifiers “computed using cryptographically strong functions directly from the digital objects… based on Merkle Acyclic Directed Graphs,” so they are an ideal optional persistent key for archived federation artefacts.

**`edges.json`** (FeatureCollection; geometry = precomputed `LineString`):

```
id          ← edgeId
source      ← fromNode
target      ← toNode
edge_type   ← edgeType   (external_reference|governance_dependency|source_archive_reference|repo_reference|data_dependency)
direction   ← derived    (directed|undirected)
weight      ← cardinality
status      ← derived    (ok|flagged; homepage-governance edges = flagged)
status_reason ← derived
layer       ← derived
min_zoom    ← derived
evidence    ← evidenceText
source_path ← evidencePath
source_line, target_line ← optional
```

**`layers.json`** (the v8 config array; each object):

```
id, label, group, type ('point'|'line'),
source_url, node_filter, edge_filter,
colour_rule, size_rule, min_zoom, max_zoom (optional),
preload (bool), visible_default (bool), description
```

**`sectors.json`** (super-node aggregation per cluster/domain):

```
id, label, bounds_abstract, parent_sector, level,
node_count, edge_count, source_url, min_zoom, summary_status
```

**`view_prefs` (localStorage only, never truth):**

```
selected_layers, pinned_nodes, collapsed_clusters,
user_order, zoom, pan, active_sector, sort_mode
```

**`manifest.json` (loaded FIRST):**

```
{
  "schema_version": "...",
  "generated_utc": "...",        // from generatedUTC
  "method_version": "...",       // from methodVersion
  "scan_id": "...",
  "counts": { "nodes": 16, "edges": 33, "sectors": N },
  "tier": "geojson" | "tiled",   // measure-then-switch
  "sources": {
     "nodes": "data/nodes.json",
     "edges": "data/edges.json",
     "layers": "data/layers.json",
     "sectors": "data/sectors.json",
     "pmtiles": null              // populated only at scale tier
  },
  "key_law_status": "UNPROVEN_ON_DECLARED_KEY",   // honest flag
  "unresolved_findings": [ "homepage_governance_edges_present" ]
}
```

### Q6 — Zoom gates (heuristics) + abstract layout assignment

Zoom gates are **tested heuristics, not fixed laws** — measure tile/byte size and adjust:

|Zoom band|What draws                                           |
|---------|-----------------------------------------------------|
|z0–2     |domains + cluster super-nodes + aggregate counts only|
|z2–4     |major repos + major edges                            |
|z4–6     |all repos in the active sector                       |
|z6–8     |workflows / contracts / reports / data products      |
|z8+      |files / evidence / commits / runs / partitions       |

Each node/edge carries a `min_zoom` derived from its `federation_level`. Implementation: MapLibre source `maxzoom` kept low and layer `minzoom` > 0 (MapLibre large-data guidance) so low zooms never attempt to render fine features.

**Layout (`x_hint`/`y_hint`) is computed at BUILD time**, two independent passes that must never be conflated:

- **Containment pass:** single-parent hierarchy from `parent_federation` → assigns each node to a cluster band / sector (vertical level = `federation_level`).
- **Dependency pass:** Sugiyama layered-DAG over the many-to-many dependency edges, run by dagre / d3-dag / ELK.js (`elk.algorithm:'layered'`). Sugiyama “places the nodes… into horizontal layers such that the edges… point in a uniform direction,”  which is exactly the dependency-direction semantics. Cycles are broken by temporary edge reversal (standard Sugiyama phase 1) and restored for display.

The two passes produce candidate coordinates that are then packed and normalised into the lon/lat window. Containment governs *which band* a node sits in; dependency governs *order within and across bands*; the two are stored as separate inputs and never overwrite each other.

### Q7 — Serving millions without loading everything (concrete tiers + thresholds)

Treat all thresholds as **measure-then-switch** heuristics:

- **Tier 0 — single GeoJSON (buildable now):** ≤ ~50k features. MapLibre handles inline/URL GeoJSON directly; the federation’s 16 nodes / ~33 edges sit here with vast headroom. Reduce coordinate precision to ~6 decimals; minify; gzip.
- **Tier 1 — sector manifests + lazy chunked JSON:** ~50k–200k features, OR whenever any single JSON exceeds a few MB. Split by sector; `manifest.json` lists per-sector URLs; fetch a sector’s JSON only when it becomes the `active_sector`. Cluster super-nodes (one aggregate node per collapsed cluster, carrying `summary_status` = worst child status) stand in at low zoom.
- **Tier 2 — client clustering:** use supercluster (the engine behind Mapbox GL clustering). On the official Mapbox engineering blog (“Clustering millions of points on a map with Supercluster,” Vladimir Agafonkin), a 400k-point dataset is indexed across all zoom levels in under a second (“total time: 972ms”) and “you can browse it smoothly at 60fps,” with a linked demo showing “6 million points loaded in the browser.” Use its property aggregation: per the mapbox/supercluster README it “supports property aggregation with the following two options: map: a function that returns cluster properties corresponding to a single point. reduce: a reduce function that merges properties of two clusters into one” — so a cluster’s status can be computed as the worst child status (the README cautions that “map must return a new object” and the reduce must not mutate its second argument). This tier kicks in around the point where low-zoom tiles would otherwise exceed budget (low hundreds of thousands of points).
- **Tier 3 — PMTiles vector tiles:** at the millions scale. Generate with tippecanoe (`-zg` to guess maxzoom; `--drop-densest-as-needed`; `tippecanoe_feature_density`), convert MBTiles→PMTiles, host the single `.pmtiles` file on GitHub Pages, register the `pmtiles://` protocol in MapLibre. Preserves “no tile server.” Set CORS `Access-Control-Allow-Headers: Range` and long immutable cache headers.

Numeric switch points (heuristics, validate on target mobile hardware): **GeoJSON < ~50k features → clustering at ~100k–250k → PMTiles beyond ~1M.**

### Q8 — SVG vs Canvas vs MapLibre/WebGL-2D vs PMTiles (element-count thresholds)

Heuristics with sources (Cylynx, 2015 MacBook; deck.gl perf docs; Memgraph):

|Renderer                        |Comfortable element budget                                    |Use when                                                                       |
|--------------------------------|--------------------------------------------------------------|-------------------------------------------------------------------------------|
|SVG                             |~1–2k nodes / ~2k edges                                       |Tiny, crisp, DOM-inspectable; the *card* UI itself (overlay) can stay SVG/HTML.|
|Canvas                          |~5k nodes / ~5k edges                                         |Mid-size, custom 2D, no DOM bloat.                                             |
|MapLibre / WebGL-2D             |~10k+ (MapLibre routinely far more via tiling)                |**Primary engine** — GPU, tiling, the v8 grammar.                              |
|deck.gl WebGL (OrthographicView)|~1M points at 60fps; ~10M degrades to 10–20fps; crash 10M–100M|Non-geospatial escape hatch at scale.                                          |
|PMTiles / vector tiles          |millions+                                                     |Static, range-request, no backend.                                             |

3D is explicitly NOT the primary UI; WebGL-2D is acceptable and expected at scale.

### Q9 — Importance, containment level, and user order as THREE independent concepts

- **IMPORTANCE** = a computed scalar driving card size/prominence. It changes *appearance*, never truth.
- **CONTAINMENT LEVEL** = position in the single-parent hierarchy (`federation_level`, `parent_federation`). Structural.
- **USER ORDER** = per-viewer arrangement (`user_order` in `view_prefs`). Cosmetic, local, never persisted as truth.

These never collapse into one another: a low-level file can be high-importance; a user can reorder cards without changing level or importance.

**`importance_score` formula (drives size only):**

```
importance_score =
    w1·in_degree            (incoming edge count)
  + w2·out_degree           (outgoing edge count)
  + w3·repo_type_weight     (ui/data > source_archive > external/unknown)
  + w4·data_criticality
  + w5·workflow_coverage    (fraction of expected workflows present)
  + w6·contract_completeness
  + w7·public_ui_dependency (does the public UI depend on it?)
  + w8·freshness            (recency of pushedAt/updatedAt)
  + w9·risk                 (e.g. flagged homepage-governance dependency)
  + w10·human_override      (explicit, logged, bounded)
  + w11·usage
  − w12·key_law_penalty     (key-law UNPROVEN lowers confidence, not truth)
  − w13·unresolved_issue_penalty
```

Normalise to [0,1], then map to radius via MapLibre `["interpolate",["linear"],["get","importance_score"],0,minR,1,maxR]`. Weights live in `layers.json`/config so they are declarative and auditable. **Changing a weight rescales cards but cannot change a node’s status, level, or evidence.**

### Q12 — Acceptance tests (checkable from a clean clone)

1. **No-backend:** loading the page issues only static-file requests (HTML/CSS/JS/JSON/optional PMTiles); zero API/server calls. *(Network panel assertion.)*
1. **Codename absent:** `grep -ri "picard\|jean-luc\|make-it-so\|starship\|command-deck\|multiverse"` over the deployed public bundle returns nothing.
1. **Manifest-first:** the app fetches `manifest.json` before any node/edge data.

1. **View-state is local + non-authoritative:** pins/order/collapse/zoom persist in localStorage and never appear in any truth file.
1. **Red can be hidden, never greened:** a flagged/red node can be toggled off but no control sets it to operational/green; grey/unknown renders as a legal state.

1. **Data integrity:** node/edge counts in `manifest.json` match the JSON payloads (16 nodes; ~30–35 deduped edges) and the source Parquet.
1. **Zoom gates:** fine layers (files/evidence) do not render below their `min_zoom`; cluster super-nodes render at z0–2.
1. **Mobile usability:** smooth pan/zoom and toggles on a mid-range phone at current scale; no freeze.
1. **Schema validity:** all JSON validates against the committed JSON Schemas in CI.

### Truth-layer vs View-layer separation (cross-cutting law)

- **Truth (user CANNOT change):** node/edge id, repo type, status, status reason, key-law status, endpoint resolution, evidence path, contract presence, workflow proof, lineage. Stored in the backend proof store (Parquet / DuckDB / audit-reports / immutable snapshots).
- **Serving layer:** `manifest.json` + layer/sector JSON (+ optional PMTiles) — generated, not authored.
- **View (user CAN change):** order, pins, collapsed/expanded, zoom, pan, active layer, sort, layout choice. Stored in localStorage only.
- **Invariant:** a red node can be *hidden* but never *made green*; grey/unknown is a legal state; the frontend makes no backend calls other than static files.

## Recommendations

**Stage 1 — Build the sandbox now (do immediately).** Execute the five steps of Q10 on the verified 16-node dataset using Tier-0 single-GeoJSON on MapLibre-as-flat-plane. This is fully buildable today and needs no scale machinery. *Benchmark:* page loads with only static requests; all 13 acceptance tests pass on a clean clone.

**Stage 2 — Harden truth/view separation and honesty (next).** Wire the JSON-Schema CI gate, the codename grep gate, and the `manifest.json` honesty flags (`key_law_status`, `unresolved_findings`). *Benchmark:* an auditor reproducing from a clean clone sees both unresolved findings surfaced, not hidden.

**Stage 3 — Add tiers only when measured thresholds are crossed.** Switch to sector-chunked lazy JSON when any payload exceeds a few MB or features exceed ~50k; add supercluster aggregation around ~100k–250k features; migrate to tippecanoe→PMTiles beyond ~1M. Keep deck.gl `OrthographicView` documented as the WebGL escape hatch if Mercator distortion or label density ever becomes objectionable. *Benchmark to trigger each switch:* measured low-zoom tile/payload byte size and sustained mobile FPS, not node count alone.

**Do NOT** (per the out-of-scope list): deploy to production, trigger workflows, add a make-it-so layer, edit repo truth from the UI, do RAG-by-click, sever the homepage dependency, reconcile the scanner key schema, make 3D primary, add P2P, rewrite data repos, or expose secrets.

## Caveats

- **The key law is not provable on the declared key.** Nodes key on `scanId|nodeId` (camelCase), not the contract’s snake_case `repo_id`. This study surfaces the gap honestly (`manifest.json` flag) but does not fix it — reconciliation is a separate future wave.
- **Homepage governance edges persist.** `governance_dependency` edges still point at `globalgrid2050-hompage`; they render as `flagged`, not removed. Severing at source is explicitly out of scope.
- **Mercator distortion is real but bounded.** Because MapLibre is locked to EPSG:3857, abstract coordinates pushed toward high latitudes stretch; confining nodes to lat ∈ [-80, 80] (and ideally nearer the equator) keeps the plane visually square. deck.gl `OrthographicView` removes this entirely if ever needed.
- **Renderer crossover numbers are directional.** The ~2k/5k/10k SVG/Canvas/WebGL thresholds come from a 2015-MacBook benchmark (Cylynx); deck.gl’s 1M/10M figures are likewise from 2015 MacBook Pros; vendors uniformly advise testing on your own graph shape and target hardware. Treat every numeric threshold here as measure-then-switch.
- **Edge dedup count is approximate.** “302 raw → ~30–35 after dedup” is from the first real scan; the exact deduped figure should be read from the build output, and `manifest.json` must reflect whatever the current export produces.
- **The federation has no final state.** Any future feature that introduces a terminal/champion node, or lets a click change status truth, would violate the audit-grade contract.
