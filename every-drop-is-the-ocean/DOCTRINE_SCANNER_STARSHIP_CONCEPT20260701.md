# Doctrine: A Recursive Repository-Scanning and Mapping System for GlobalGrid2050

## TL;DR

- **Build the recursive “starship” scanner in three staged layers, permissive-static first.** Start with a declared-only per-repo cartridge (SBOM + manifest dependencies transcribed faithfully via Syft/cdxgen and the GitHub dependency-submission API), then add a separate PROV-marked “derived” overlay for inferred call/data-flow edges (Tree-sitter as the transcription-grade parser; stack-graphs/SCIP for name resolution), then escalate rendering from inline-JSON radial → PMTiles vector tiles → WebGL graph engines only as node counts demand. This honors transcribe-don’t-embellish because declared facts and inferred facts live in physically separate, provenance-stamped layers.
- **The doctrine’s own laws map cleanly onto known standards.** Content-address every source artifact an edge points to with SWHID (now ISO/IEC 18670:2025); mark every inferred edge with W3C PROV-O `wasDerivedFrom`/`wasGeneratedBy`; keep each recursion level’s cartridge independently valid on its declared key with DuckDB read-back; empty/unparseable repos degrade to a valid empty cartridge (zero nodes, zero edges, valid manifest).
- **Stay strictly permissive-and-static as long as you can; a backend is justified only at true millions-of-nodes-with-full-call-graphs scale.** DuckDB + Parquet (+ DuckDB-WASM in the browser) is the queryable proof store at every level. If graph queries are needed, prefer DuckPGQ (SQL/PGQ on your existing Parquet, but note it is a research-stage community extension) over standing up a graph database. Do NOT build on Kùzu (MIT; archived 10 October 2025 after Apple’s acquisition, with v0.11.3 as its final release). Isolate or avoid copyleft/proprietary tools (Joern is Apache-2.0 but heavy; CodeQL is proprietary and OSS-only-licensed; igraph/OGDF are GPL; yFiles/Ogma are commercial).

## Key Findings

1. **Declared vs. inferred is a hard, checkable line.** Imports listed in package.json/requirements.txt/go.mod/Cargo.toml, and the dependency lists produced by SBOM tools, are *transcribed declarations* — they belong in the base cartridge. Call graphs, data-flow edges, and symbol-reference edges produced by static analysis are *inferred* — they must live in a separate derived overlay with provenance. Tree-sitter produces a concrete syntax tree (a faithful transcription of what the file literally says); the moment you resolve a name across files or infer a call target, you have crossed into inference.
1. **Every time-sensitive fact in the brief checks out.** SWHID was adopted as international standard ISO/IEC 18670:2025 on 23 April 2025 (ISO listing 89985, Edition 1, “SWHID Specification V1.2”). Kùzu’s GitHub repo was archived 10 October 2025 — the same day v0.11.3 shipped as a final release with a note that the team was “working on something new”; Apple agreed to acquire Kùzu Inc. on 9 October 2025 (disclosed via an EU Digital Markets Act filing; the EU list records it as “October 10, Kuzu”). Kùzu (MIT-licensed) was founded 2023 in Ontario by Semih Salihoğlu with ~10 employees. cosmos.gl (created by Nikita Rokotyan, maintained by Rokotyan and Olya Stukova) entered the OpenJS Foundation’s Incubation Phase, with the goal of graduating to an At-Large Project. GridCal was renamed VeraGrid after a company registered the “GridCal” trademark and threatened suit, and was relicensed from LGPL to MPL-2.0 (Issue #300); it is now maintained by eRoots Analytics. DuckPGQ is a real community extension implementing SQL/PGQ (SQL:2023) but is explicitly a research project / work in progress.
1. **The three-tier rendering escalation is real and has concrete crossover points.** Inline JSON/GeoJSON radial is fine to tens of thousands of features. PMTiles (single-file, static, BSD-3) built by tippecanoe handle millions of features  served straight from GitHub Pages. WebGL graph engines (sigma.js, deck.gl, cosmos.gl) handle 10^5–10^6-node interiors. Mobile GPUs are the fragile link: cosmos.gl depends on float-texture WebGL extensions that have broken on iOS and are missing on some Android devices. 

## Details

### 1. Code-structure extraction tooling (permissive-first)

The transcription-grade base of the code layer is **Tree-sitter** (MIT), an incremental parser with a uniform C API  and grammars for dozens of languages; its output (a concrete syntax tree) is a faithful transcription  and it runs statically with no backend. It is the correct foundation because GitHub itself uses it, and it is dependency-free (pure C runtime).

For symbol indexes, **universal-ctags** produces a tag index but is GPL-2.0-or-later (copyleft — isolate as a subprocess, do not link into a permissive core). **SCIP** (Sourcegraph’s Code Intelligence Protocol, a Protobuf schema; successor to **LSIF**) is a standardized cross-language symbol/reference format emitted by scip-typescript, scip-python, scip-java, rust-analyzer, and others; it is the honest interchange format for *name resolution* facts — but note that resolved references are inferred, not declared. **stack-graphs** (github/stack-graphs, Apache-2.0/MIT dual, built on Tree-sitter) does file-incremental name resolution without a build system and has run in GitHub production since November 2021; it is the best permissive path to derived symbol edges at scale.

For deeper call/data-flow analysis, the **Code Property Graph** (CPG = AST + CFG + PDG merged)  is the reference concept. **Joern** implements it and is Apache-2.0 (open) — but it is a heavy Scala/JVM platform that stores CPGs in its own graph database and is built for security analysis at cost; treat it as an optional, isolated derived-overlay generator, not core. Its output is entirely inferred. **CodeQL must be avoided for the permissive core**: its CLI/engine is proprietary (the “GitHub CodeQL Terms,” cataloged as `github-codeql-terms-2020`; only the *queries* in github/codeql are MIT). Its license restricts use to open-source codebases plus academic/demo purposes — verbatim, it forbids using the software “in connection with any codebase that is not an Open Source Codebase (e.g., code in a private repo in GitHub)” and states “CodeQL can’t be used for automated analysis, continuous integration or continuous delivery… unless you have a license for GitHub Advanced Security.”

Language-native dependency graphers: **madge** (MIT, JS/TS module graphs) and **dependency-cruiser** (MIT, JS/TS/multi-language, validation rules) for JavaScript; **pydeps** (BSD-2-Clause), **pipdeptree** (MIT), and Google’s **importlab** (Apache-2.0, archived) for Python; **jdeps** (ships with the JDK/OpenJDK, GPLv2-with-Classpath-Exception — copyleft but linking-exempt) for Java; **go mod graph** (part of the Go toolchain, BSD-3) for Go; **cargo tree** (part of Cargo, MIT/Apache-2.0 dual) and **cargo-modules** (MPL-2.0, weak copyleft) for Rust. Module/import edges these emit from manifests are *declared*; anything they infer about the actual call structure is *derived*.

### 2. The SBOM / dependency-declaration layer (the transcribed base)

**Syft** (Anchore, Apache-2.0) generates SBOMs in both SPDX and CycloneDX from source trees and containers across many ecosystems.  **cdxgen** (OWASP, Apache-2.0) is CycloneDX-focused, covers 20+ languages, and can invoke build tools for accurate transitive graphs.  Both transcribe *declared* dependencies — this is the honest base beneath any inferred call graph. GitHub’s **dependency graph / dependency-submission API** exports an SPDX SBOM and accepts submitted snapshots (a commit-SHA-anchored dependency set); it is the zero-infrastructure way to get a declared per-repo dependency cartridge. (Caution: a March 2026 report flagged Trivy as compromised in supply-chain attacks;  prefer Syft/cdxgen.) The per-repo dependency cartridge should transcribe these lists verbatim, mark them `declared`, and content-address the manifest file each edge came from.

### 3. Glide / dive / surface traversal and scale architecture

The recursion (federation → repo → module → file → symbol) should be **lazy and sharded**: generate a child cartridge per repo on demand rather than cloning a million repos at once. Use **metadata-only probing** (GitHub API, canonical-file existence) for the federation and repo-summary levels, and only **full-clone parse** a repo when a user dives into it (or in a batched CI pass for popular repos). Cache aggressively with **content-addressed cartridges**: SWHID (built on Merkle Acyclic Directed Graphs, ISO/IEC 18670:2025) gives you intrinsic identity for dedup and change detection — re-scan a repo only when its SWHID changes. Stack-graphs’ file-incremental model is the same principle at file granularity.

**DuckDB + Parquet** is the queryable proof store at each level; **DuckDB-WASM** runs the same engine in-browser, querying remote Parquet via HTTP range requests (reading only the row groups needed — a `count(*)` can finish on file metadata alone). Partition the store **Hive-style by scope/repo** (`scope=.../repo=.../nodes.parquet`) so a dive loads only the relevant child cartridge and predicate pushdown skips the rest.  Caveat: DuckDB-WASM lacks full feature parity with native DuckDB  and the binary is large (tens of MB — the “eh” build is ~33 MB), which hurts first-load — lazy-load it only when a dive needs interactive querying.

**The empty case is first-class.** If a repo has nothing to map (empty, binary-only, or unparseable), it renders as a **valid empty cartridge**: a well-formed manifest with zero nodes and zero edges that still passes DuckDB read-back on its declared key. “If there is nothing then fine” is enforced by making empty a valid state, not an error.

### 4. Rendering at depth and scale

Three tiers, escalate only as counts demand:

- **Tier 1 — inline JSON/GeoJSON radial** (the current `federation_radial.html`). Good to tens of thousands of features. Keep as default for federation and small repos.
- **Tier 2 — PMTiles vector tiles** built by **tippecanoe** (developed by Erica Fischer at Felt; PMTiles reference implementation is BSD-3-Clause). A single static file on GitHub Pages / S3 serves millions of features  with level-of-detail baked in; MapLibre GL renders it. This is the scale-out for huge interiors while staying static.
- **Tier 3 — WebGL graph engines** for dense 10^5–10^6-node graph interiors: **sigma.js** (MIT, WebGL, practical ceiling ~100k–500k nodes, renders a graphology model), **deck.gl** (WebGL2/WebGPU, layer-based, used by Microsoft’s MSAGLJS tile-pyramid graph renderer),  **cosmos.gl** (OpenJS incubating, GPU force layout, >1M nodes). 

**Cytoscape.js** (MIT — relicensed from LGPL) with **compound nodes + expand/collapse** is the right recursion/drill-down UX model — it maps directly onto the child_manifest scope model. Use **level-of-detail / semantic zoom** so diving into a repo *swaps cartridges* rather than rendering everything at once.

**Mobile-GPU caveats (be honest):** cosmos.gl requires float-texture WebGL extensions (OES_texture_float on Android; EXT_float_blend for its Many-Body force on iOS) — it broke on iOS 15.4 and does not work on Android devices lacking OES_texture_float.  The practical implication: the SVG→Canvas→WebGL→vector-tile crossover arrives *earlier* on phones, so prefer PMTiles (Tier 2) for mobile-facing deep maps and reserve WebGL force layouts (Tier 3) for desktop.

### 5. Two architecture branches

**(a) Strictly permissive + static (RECOMMENDED default).** GitHub Pages + Actions + DuckDB/Parquet/PMTiles + Tree-sitter/stack-graphs/Syft, all MIT/BSD/Apache. Scales impressively: metadata-only federation maps, per-repo declared cartridges, and PMTiles interiors all serve from static storage with no backend. It breaks when you need *interactive cross-repo graph queries over millions of inferred edges* (e.g., “show every call path from any repo into this function”) — DuckDB-WASM’s binary size and range-request chattiness, plus browser memory, become the wall.

**(b) Backend / graph-DB-assisted (only if the ambition demands it).** If you genuinely need million-node, all-the-way-to-raw-code cross-repo traversal, the lightest honest step is **DuckPGQ** (SQL/PGQ over your *existing* Parquet store — no new datastore, no ETL) — but it is a research-stage community extension (“still under active development”), so treat it as experimental and keep the Parquet base authoritative. A full graph database is a larger commitment: **do not build on Kùzu** (archived October 2025  after Apple’s acquisition). Joern’s embedded CPG store is an option only for isolated deep-analysis jobs. The cost of any backend is auditability: the doctrine’s “declared checks on the real key with DuckDB read-back” is easiest to guarantee when the proof store *is* the queryable store (Parquet/DuckDB), so crossing into a backend means re-proving that invariant against the new engine.

**Recommendation:** stay in branch (a); adopt DuckPGQ (still branch-a-flavored, since it runs on Parquet) only when a concrete query need appears; cross to a real backend only when a measured requirement (interactive million-edge cross-repo path queries) cannot be met statically.

### 6. Governance / honesty mechanics

- **Per-edge declared-vs-inferred flag.** Every edge row carries a `provenance` enum: `declared` (transcribed from a manifest/lockfile) or `derived` (inferred by static analysis). Derived edges live in a *separate overlay* Parquet, never mixed into the transcribed base — this is the mechanical enforcement of transcribe-don’t-embellish and separation-of-duties.
- **W3C PROV-O for the derived overlay.** Each derived artifact is a `prov:Entity` linked by `prov:wasDerivedFrom [![](claude-citation:/icon.png?validation=0D0D2271-AA1F-47D0-9705-0DC136A75227&citation=eyJlbmRJbmRleCI6MTM3MDksIm1ldGFkYXRhIjp7Imljb25VcmwiOiJodHRwczpcL1wvd3d3Lmdvb2dsZS5jb21cL3MyXC9mYXZpY29ucz9zej02NCZkb21haW49dzMub3JnIiwicHJldmlld1RpdGxlIjoiUFJPVi1POiBUaGUgUFJPViBPbnRvbG9neSIsInNvdXJjZSI6IlczQyIsInR5cGUiOiJnZW5lcmljX21ldGFkYXRhIn0sInNvdXJjZXMiOlt7Imljb25VcmwiOiJodHRwczpcL1wvd3d3Lmdvb2dsZS5jb21cL3MyXC9mYXZpY29ucz9zej02NCZkb21haW49dzMub3JnIiwic291cmNlIjoiVzNDIiwidGl0bGUiOiJQUk9WLU86IFRoZSBQUk9WIE9udG9sb2d5IiwidXJsIjoiaHR0cHM6XC9cL3d3dy53My5vcmdcL1RSXC9wcm92LW9cLyJ9XSwic3RhcnRJbmRleCI6MTM2OTAsInRpdGxlIjoiVzNDIiwidXJsIjoiaHR0cHM6XC9cL3d3dy53My5vcmdcL1RSXC9wcm92LW9cLyIsInV1aWQiOiIwOTA3ZGFlNC1jZWYzLTQ3MjEtODg4ZC1jMWJmMTJjMmU4ODcifQ%3D%3D "W3C")](https://www.w3.org/TR/prov-o/)` to the source artifacts and `prov:wasGeneratedBy` to the analysis activity (with tool name + version); - **SWHID content addressing.** Every edge’s `evidencePath` resolves to a SWHID (ISO/IEC 18670:2025) pinning the exact file/blob/tree the evidence points to — computed intrinsically by anyone with a copy, no registry needed.
- **Declared checks extend to child cartridges.** Keep each level valid on its real key (nodes: `scanId+nodeId`; edges: `scanId+fromNode+toNode+edgeType+evidencePath`) with DuckDB read-back. Add referential checks: every child edge resolves to a real child node; every parent `child_manifest` pointer resolves to a valid child cartridge that itself passes its own declared checks. Reproducible from a clean clone: the whole map rebuilds deterministically from source + pinned tool versions.

### Licensing-trap list (isolate or avoid)

- **Proprietary:** CodeQL CLI/engine (GitHub CodeQL Terms; OSS-only, needs GitHub Advanced Security for non-OSS/automated use). yFiles, Ogma, KeyLines, Tom Sawyer — commercial graph SDKs.
- **GPL / strong copyleft (isolate as subprocess, do not link into permissive core):** igraph (GPL-2+), OGDF (GPL-2/3), universal-ctags (GPL-2+), Pandoc (GPLv2+), NodeBox/statnet/multiNetX (GPL).
- **Weak/file-level copyleft (usable with care, keep files separate):** cargo-modules (MPL-2.0), Graphviz (EPL-2.0 — note: changed from CPL-1.0, it is EPL now, not CPL), jdeps (GPLv2+Classpath Exception — linking-exempt), VeraGrid (MPL-2.0).
- **Archived / do-not-build-on:** Kùzu (MIT; archived Oct 2025 after Apple acquisition, v0.11.3 final). importlab (archived, still Apache-2.0).

## Recommendations

**Stage 1 — Declared-only per-repo cartridge (ship first).** Extend the existing scanner to emit, per repo, a base cartridge whose edges are *only* declared dependencies: parse manifests (package.json/requirements.txt/go.mod/Cargo.toml) and/or run Syft/cdxgen, transcribe verbatim, mark every edge `declared`, content-address each evidence file with SWHID, and validate on the real key with DuckDB read-back. Empty repos → valid empty cartridge. **Threshold to proceed:** base cartridges validate for 100% of repos (including empties) and render at Tier 1.

**Stage 2 — Provenance-marked derived overlay.** Add Tree-sitter parsing (transcription-grade CST) plus stack-graphs/SCIP for name resolution and, where justified, a Joern job for call/data-flow. Write these as a *separate* derived Parquet overlay, every edge stamped `derived` with PROV-O `wasDerivedFrom`/`wasGeneratedBy` (tool + version) and SWHID evidence. The UI renders the base by default and the derived overlay as a toggle, visually distinct. **Threshold to proceed:** derived overlay never contaminates the base (audit passes); every derived edge is attributable and reproducible from a clean clone.

**Stage 3 — Scale rendering.** Keep Tier 1 for federation/small repos. When a cartridge exceeds ~tens of thousands of features, generate Tier 2 PMTiles via tippecanoe (static, GitHub Pages). Reserve Tier 3 WebGL (sigma.js / deck.gl / cosmos.gl, with Cytoscape.js compound-node drill-down) for dense desktop interiors. **Threshold to escalate a given repo:** feature count crosses the tier boundary; **threshold to add a backend:** a measured, recurring need for interactive cross-repo path queries over millions of inferred edges that DuckDB-WASM + DuckPGQ cannot serve — at which point adopt DuckPGQ first, and only then a graph DB (never Kùzu).

**Governance guardrails (all stages):** declared and derived in physically separate layers; per-edge provenance flag mandatory; SWHID on every evidence pointer; DuckDB read-back on the real key at every recursion level; child_manifest pointers must resolve to valid child cartridges; the whole map must rebuild deterministically from a clean clone with pinned tool versions.

## Caveats

- **DuckPGQ, cosmos.gl, and importlab maturity:** DuckPGQ is explicitly a research project / community extension (“still under active development”); cosmos.gl is an OpenJS *incubating* (not yet graduated) project; importlab is archived. Pin versions and treat as experimental.
- **Inference is never ground truth.** Call graphs and data-flow edges are approximations (dynamic dispatch, reflection, monkey-patching defeat static analysis). The `derived` flag and PROV provenance are not decoration — they are the honest admission that these edges are computed, not observed.
- **Mobile rendering is genuinely fragile.** Float-texture WebGL extension gaps on iOS/Android mean WebGL force layouts can silently fail; PMTiles is the safer deep-map path on phones.
- **Trivy supply-chain compromise** was reported (systemshardening / sbomify, March 2026) — this is a secondary-source claim; verify current status before adopting Trivy. Prefer Syft/cdxgen regardless.
- **Licenses change.** GridCal→VeraGrid (LGPL→MPL-2.0)  and Cytoscape.js (LGPL→MIT) show that license status is time-sensitive; re-verify any copyleft/proprietary flag at build time. Graphviz moved from CPL-1.0 to EPL-2.0.
- **Graph-DB landscape is consolidating.** Kùzu’s acquisition/archival is a warning: do not hard-couple to any single embedded graph engine. Keeping the Parquet/DuckDB proof store authoritative insulates you from any one engine’s fate.
- **jdeps license nuance:** it inherits the OpenJDK license (GPLv2+Classpath Exception); a non-OpenJDK JDK (e.g., Oracle’s under its No-Fee Terms) carries different terms — footnote this if your CI uses a vendor JDK.
