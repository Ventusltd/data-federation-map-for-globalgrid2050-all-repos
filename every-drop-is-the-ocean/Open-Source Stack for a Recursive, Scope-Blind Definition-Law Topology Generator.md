# Open-Source Stack for a Recursive, Scope-Blind Definition-Law Topology Generator

## TL;DR

- **Keep MapLibre GL JS as the baseline 2D renderer, but treat it as the small/medium tier of a three-tier rendering strategy: inline GeoJSON (≤ ~50k features) → PMTiles vector tiles (millions) for the map plane, with deck.gl (OrthographicView + ScatterplotLayer/LineLayer/PathLayer) and cosmos.gl/sigma.js as WebGL overlays/alternatives when you must draw hundreds of thousands to millions of nodes.** Every recommended tool in the core path is MIT/BSD/Apache and works from static GitHub Pages with no backend.
- **The architecture’s central insight — that the topology is already authored by each authority — maps cleanly onto a permissive open-source stack:** SBOM/dependency tools (Syft, cdxgen, madge, dependency-cruiser) for software; PowSyBl/pandapower/PyPSA/CIM-CGMES for power; GTFS for transit; KiCad/kiutils for PCB; Pandoc/unified for documents — each *transcribes* an existing declaration rather than inferring one. The cartridge contract (manifest + nodes + edges + layers + sectors + child_manifest) is essentially a graph-specialized Frictionless **Data Package** (`datapackage.json` + resources), which you should adopt as the schema backbone, validated with JSON Schema and provenance-stamped with W3C PROV.
- **Recommended build order: (1) build the scope-blind engine against a federation-scope cartridge first; (2) write the repo-internals scanner (madge/dependency-cruiser/Syft → cartridge) to prove recursion via `child_manifest`; (3) add domain scanners (power, transit, PCB, docs).** Biggest risks are (a) layout/inference creep where a scanner *embellishes* topology the authority never declared, (b) the SVG→Canvas→WebGL→vector-tile crossover being hit earlier than the headline numbers on mobile GPUs, and (c) a few copyleft traps (ELK is EPL, GridCal is now MPL-2.0 but legacy LGPL, OGDF is GPL).

## Key Findings

1. **MapLibre GL JS (BSD-3) is correct for the map plane but is not a million-edge graph renderer.** Its GeoJSON source converts to vector tiles in a web worker via geojson-vt — whose README demonstrates “geojson-vt action in MapLibre GL JS, dynamically loading a 100Mb US zip codes GeoJSON with 5.4 million points,” so the engine itself can pre-tile large data. Community practice nonetheless puts the *inline*-GeoJSON ceiling at roughly 50k features before you should pre-tile out of band. Past that, switch to **PMTiles** (BSD-2) — a single-file vector-tile archive served by HTTP range requests directly from GitHub Pages — generated with **tippecanoe** (BSD). This is the canonical no-backend path to millions of features.
1. **For true large-graph rendering, deck.gl (MIT) renders ~1M items at 60 FPS and degrades to 10–20 FPS near 10M; cosmos.gl (MIT, an OpenJS Foundation incubating project as of May 2025) does GPU force layout AND rendering of over one million nodes/links in-browser and pairs natively with DuckDB; sigma.js+graphology (MIT) is the mature choice for tens-of-thousands to low-hundreds-of-thousands.** deck.gl’s OrthographicView is the non-geospatial flat-plane mode that fits the “abstract coordinates in lon/lat slots” design.
1. **The cartridge contract is almost identical to a Frictionless Data Package** (`datapackage.json` “manifest + resources” model). Adopt Data Package conventions and validate every cartridge with **JSON Schema**. Stamp provenance with **W3C PROV-O** (entities/activities/agents) and, for build pipelines, **OpenLineage**. Content-address cartridges with Merkle hashing; use **SWHID** (standardized as ISO/IEC 18670:2025 on 23 April 2025) for source artifacts.
1. **Build-time layout: use Graphviz sfdp for large undirected graphs, ELK/elkjs (EPL-2.0) for layered/Sugiyama + compound/hierarchical/ports, dagre/d3-dag for simpler DAGs, and igraph/NetworkX/cuGraph for analytics-scale.** OGDF is GPL — avoid unless you isolate it as a build-only CLI.
1. **Power systems is the strongest “already-declared” case: pandapower (BSD-3) exports a NetworkX node/edge graph directly via `topology.create_nxgraph()` respecting switch states; PowSyBl (MPL-2.0, LF Energy) literally generates single-line diagrams and network-area diagrams; PyPSA (MIT) is bus/branch native; CIM/CGMES (IEC 61970/61968, ENTSO-E) is the authority’s RDF/XML declaration.** GridCal recently re-licensed to MPL-2.0.
1. **Cytoscape.js (MIT) is uniquely relevant to recursion** because of its first-class compound/nested nodes and `expand-collapse` extension  — a good model for the Russian-doll behavior, though for very large graphs you’ll still render via WebGL libraries.

## Details

### Layer 1 — Rendering engine for very large 2D topologies

**MapLibre GL JS (BSD-3-Clause)** — Already chosen; correct for a flat/non-geographic plane because it natively uses lon/lat coordinate order and you can place abstract coordinates in those slots. Internally a GeoJSONSource is sliced into vector tiles by **geojson-vt** inside a web worker; the geojson-vt README explicitly shows it “dynamically loading a 100Mb US zip codes GeoJSON with 5.4 million points” inside MapLibre. The official “large data” guidance is to reduce coordinate precision, strip unused properties, set layer `minzoom`, cluster points (via supercluster), and split sources. **Honest limit:** inline GeoJSON works well into the tens of thousands of features; multiple community practitioners pre-process to vector tiles (tippecanoe → MBTiles/PMTiles) once a dataset exceeds roughly 50k points or ~45 MB, because MapLibre re-tiles GeoJSON on the main data path and each source carries high per-source overhead. **Fit:** ideal engine for the map metaphor, sectors (clusters), and layers.json (declarative style layers map 1:1 to MapLibre style layers). Switch to vector tiles when a single scope’s nodes/edges exceed ~50k.

**deck.gl (MIT, OpenVisualization/Linux Foundation)** — GPU-accelerated layer framework. Official performance guidance is precise: *“On 2015 MacBook Pros with dual graphics cards, most basic layers (like ScatterplotLayer) renders fluidly at 60 FPS during pan and zoom operations up to about 1M (one million) data items, with framerates dropping into low double digits (10-20FPS) when the data sets approach 10M items”* — and Chrome’s 1 GB per-allocation cap crashes most layers between 10M and 100M items. For graphs: **ScatterplotLayer** (nodes), **LineLayer/PathLayer** (edges), and the **GraphLayer** example (an adaptor pattern linking a d3-force layout to the renderer). **OrthographicView** gives a non-geospatial 2D plane — the right mode for abstract coordinates. deck.gl can also be layered as a MapLibre custom layer, so you can overlay GPU graph rendering on the MapLibre plane. **Fit:** the high-scale renderer for a single scope’s interior when node counts run 10^5–10^6.

**cosmos.gl / Cosmograph (MIT; cosmos.gl joined the OpenJS Foundation as an incubating project, announced 14 May 2025)** — GPU force-directed layout AND rendering, all in WebGL shaders, avoiding CPU↔GPU transfer. The OpenJS announcement states cosmos.gl “enables scalable and performant visualization of over one million nodes and links,” created by Nikita Rokotyan and maintained by Nikita and Olya Stukova. Cosmograph’s stack pairs with DuckDB(-WASM) and SQL for filtering. **Maturity:** cosmos.gl v3.0 ported rendering from regl to **luma.gl (WebGL 2)**; actively developed. **Mobile caveat to flag (from the cosmos.gl README):** “cosmos.gl doesn’t work on Android devices that don’t support the OES_texture_float WebGL extension,” and it has historically depended on float-blend WebGL extensions on iOS — test on target devices. **Fit:** best-in-class when you need *both* layout and rendering of a very large scope at runtime and don’t want a build-time layout step.

**sigma.js + graphology (MIT)** — Purpose-built WebGL graph renderer; graphology supplies the data model + algorithms (ForceAtlas2, community detection, metrics). Self-described target: *“graphs of thousands of nodes and edges”*; in practice comfortable into the tens of thousands and beyond. Reads GEXF directly (Gephi interchange). **Fit:** mature, well-documented middle tier; good for repo-scope and substation-scope interiors. Used in production tools (e.g., the Helveg software-documentation diagrams, which modify Sigma’s WebGL shaders).

**regl / PixiJS / ngraph (VivaGraphJS, ngraph.pixel) by Andrei Kashcha** — Lower-level WebGL (regl) and 2D WebGL scene graph (PixiJS, MIT) building blocks; ngraph is a fast graph data-structure + layout family. These are the “roll-your-own renderer” options; ngraph/VivaGraphJS demonstrably handle tens of thousands of nodes. **Fit:** only if you need a bespoke renderer; otherwise deck.gl/sigma/cosmos cover the space with less effort.

**AntV G6 (MIT) and Cytoscape.js (MIT)** — G6 is a full-featured graph-viz framework (Canvas/SVG/WebGL). **Cytoscape.js is the standout for recursion:** it natively supports **compound (nested) nodes** via the `parent` field, with **CoSE/fCoSE** compound layouts and an **expand-collapse** extension — a direct conceptual match for the Russian-doll / `child_manifest` drill-down. Honest scale: Cytoscape.js is Canvas-based and best below ~5–10k elements; it is a *model for the recursion UX*, not the million-node renderer. yFiles is the commercial gold standard but is **not open-source** — excluded.

**Honest crossover thresholds (SVG → Canvas → WebGL → vector tiles):**

- **SVG (d3, Cytoscape SVG):** clean, fully stylable, interactive — but fails animating beyond a few thousand elements. Use ≤ ~2–5k.
- **Canvas (Cytoscape.js, G6 canvas):** ~5k–50k elements with acceptable interactivity.
- **WebGL (deck.gl, sigma.js, cosmos.gl, PixiJS):** ~10^4–10^6 elements; cosmos.gl/deck.gl push to ~1M, degrading near 10M (and crashing 10M–100M in Chrome due to the 1 GB allocation cap).
- **Vector tiles (MapLibre + PMTiles):** the only path to *millions* of features as static files, because the client fetches only the byte ranges for the current viewport. This is the right tier whenever a single scope’s declared topology is genuinely massive (national grids, monorepo dependency universes).

### Layer 2 — Large-scale tiled serving as static files (no backend)

**PMTiles (BSD-2-Clause; Brandon Liu/Protomaps)** — A single-file archive of Z/X/Y tiles with an embedded directory; per the Protomaps docs, “PMTiles readers use HTTP Range Requests to fetch only the relevant tile or metadata inside a PMTiles archive on-demand,” so it hosts on “any static file server (S3, Cloudflare R2, GitHub Pages)” with no tile server. The current spec is **Version 3**. It is read-only: “It is not possible to update an archive in-place without re-writing the entire file, similar to CSV, JSON and Parquet.” Integrates with MapLibre via the `pmtiles://` protocol plugin. Production reports include a ~900 MB Australia-only OSM extract served from S3. **Fit:** the canonical no-backend serving tier for the map plane at scale. CORS + `Range` headers and long cache-control are the operational gotchas to flag.

**tippecanoe (BSD; felt/tippecanoe)** — Generates vector tiles (MBTiles/PMTiles) from GeoJSON at scale, with zoom-dependent simplification and feature dropping. The standard generator feeding PMTiles. (Planetiler and GDAL/ogr2ogr are alternatives.) **Fit:** build-time step converting each scope’s nodes.json/edges.json into tiles when the scope is large.

**Parquet + DuckDB-WASM (both MIT)** — DuckDB-WASM runs a columnar SQL engine entirely in-browser, querying Parquet via **HTTP range requests with predicate pushdown + column pruning + Hive-partition pruning**, so a client can scan only the relevant row groups of a remote file with no backend. This is exactly the “DuckDB/Parquet proof store” already named in the design. **Fit:** the proof/attribute store and the query layer behind both the engine (find-a-node, filter-by-attribute) and per-scope honesty findings. Hive-partition by scope to enable cheap drill-down.

**GeoParquet / Apache Arrow (Apache-2.0)** — GeoParquet standardizes geometry-in-Parquet; Arrow is the zero-copy in-memory interchange between DuckDB results and deck.gl (deck.gl has Arrow layers). **Fit:** store nodes/edges as (Geo)Parquet for the proof store; hand query results to deck.gl as Arrow with minimal copying.

**Tiering recommendation:** inline GeoJSON in the cartridge for small scopes (≤ ~50k features) → chunked JSON or DuckDB/Parquet queries for medium → PMTiles vector tiles for millions. The cartridge `manifest.json` should declare which tier each resource uses so the engine stays scope-blind.

### Layer 3 — Build-time graph layout (assign abstract x/y)

- **Graphviz (permissive EPL/CPL-style license):** `dot` (layered), `neato`/`fdp` (force), and **`sfdp`** — “a fast multilevel force directed algorithm that efficiently layout very large graphs,” with effective PRISM node-overlap removal; handles ~1k–~100k+ nodes (needs the GTS triangulated-surface build for best results). The classic, scriptable, build-time choice. Related research tooling like `mars` extends stress-majorization layout to “several hundred thousand nodes.”
- **ELK / elkjs (EPL-2.0):** the **Eclipse Layout Kernel** — flagship **ELK Layered** (Sugiyama, 140+ options), plus force, stress, tree, radial, and rectangle-packing. Crucially supports **compound/hierarchical graphs and ports** — the best fit for hierarchical/recursive layouts and for SLD-like port-aware diagrams. elkjs is the transpiled JS build (runs at build time in Node or in-browser). **License flag:** EPL-2.0 is a weak copyleft; fine to *use* as a library/tool but note the obligation if you modify ELK itself.
- **dagre / d3-dag (MIT):** lightweight layered-DAG layout in pure JS; great for dependency DAGs at modest scale; simpler than ELK.
- **OGDF (GPL):** C++, scales to very large graphs with high-quality algorithms — **but GPL**; only use as an isolated build-time binary, never linked into distributed code.
- **igraph (GPL-2.0), NetworkX (BSD), graph-tool (LGPL/GPL mix), SNAP (BSD):** analytics + layout. NetworkX (BSD) is the safe permissive default for build-time analysis and integrates directly with pandapower’s NetworkX graph export. igraph is faster but GPL.
- **cuGraph / RAPIDS (Apache-2.0):** GPU graph analytics + ForceAtlas2 layout for genuinely huge graphs at build time (needs an NVIDIA GPU). **ForceAtlas2** also exists in graphology (JS) and Gephi.

**Embellishment flag:** layout is the single biggest place the system can *lie*. Coordinates are visualization, not declared truth. The contract already separates this correctly by putting abstract coordinates in nodes.json. **Rule: if the authority declared positions (e.g., CGMES DiagramLayout, KiCad footprint coordinates, a transit feed’s shapes.txt), the scanner must transcribe them; only when the authority declared *no* geometry may the build step compute a layout — and that computed layout must be marked as derived in provenance.**

### Layer 4 — Domain scanners (read the already-declared definition)

**Software / repositories:**

- **Dependency/SBOM tier:** **Syft (Apache-2.0, Anchore)** generates SPDX *and* CycloneDX from source trees and images; **cdxgen (Apache-2.0, OWASP)** is the official CycloneDX generator across 20+ languages and emits an explicit `dependencies` array (component-ref → dependents) — a ready-made edge list. **SPDX** (ISO/IEC 5962:2021) uses an explicit relationship graph (DEPENDS_ON, CONTAINS); **CycloneDX** (ECMA-424) keeps a flat component list + separate dependency graph. GitHub’s dependency-graph / dependency-submission API is the authority’s own declaration for hosted repos.
- **Language import graphs (transcription, not inference):** **madge (MIT)** and **dependency-cruiser (MIT)** for JS/TS (both emit DOT/JSON and detect cycles); pydeps/pipdeptree for Python; `jdeps` for Java; `go mod graph`; `cargo tree`. These read declared imports/manifests.
- **Deeper structure (use with care — embellishment risk):** **Tree-sitter (MIT)** parses source to ASTs/symbol graphs; **universal-ctags** for symbol indexes; **Joern / Code Property Graph (Apache-2.0 spec; the 2014 Yamaguchi et al. CPG paper won the 2024 IEEE Test-of-Time Award)** merges AST+CFG+PDG; **CodeQL** (proprietary engine, restrictive license — avoid for a permissive stack). **Flag:** AST/CPG tools can *infer* call/data-flow edges the authority never “declared.” For a transcription-faithful scanner, prefer the manifest/import level (madge, Syft) and treat CPG-derived edges as a separate, provenance-marked overlay.

**Power systems / load flow / SLD (the strongest already-declared case):**

- **pandapower (BSD-3, Fraunhofer IEE/Univ. Kassel):** pandas element tables. Its `topology.create_nxgraph()` “converts a pandapower network into a NetworkX graph … Busses are being represented by nodes … edges represent physical connections between buses (typically lines or trafos),” and “open switches … are being considered (no edge between nodes)” — a turnkey, switch-aware scanner output. Imports MATPOWER, PSS/E, PowerFactory, **CIM CGMES 2.4.15/3.0**, UCTE-DEF.
- **PowSyBl (MPL-2.0, LF Energy):** Java; the **single-line-diagram** library literally generates SLD SVGs (node/breaker + bus/breaker topology, multiple layout modes incl. CGMES-DL), and the **network-area-diagram** “displays a graph whose nodes are the network voltage levels, and whose edges are the lines and transformers between those voltage levels.” IIDM model serializes to XIIDM/JIIDM/BIIDM; imports CGMES, UCTE, IEEE-CDF, Matpower, PSS/E. **The exemplar open-source SLD generator.**
- **PyPSA (MIT, TU Berlin):** bus/branch native (`buses`, `lines`, `transformers` DataFrames; `determine_network_topology()`); datasets PyPSA-Eur and PyPSA-GB are MIT.
- **GridCal (MPL-2.0 on master; some PyPI releases still tagged LGPL; project being renamed VeraGrid):** imports PSS/E raw/rawx, Matpower, CIM 16, CGMES 2.4.15/3.0, DGS, EPC, PyPSA; internal MultiCircuit bus/branch model; exports gridcal/xlsx/json, CGMES, PSS/E.
- **OpenDSS (BSD-3, EPRI):** distribution-feeder `.DSS` scripts; no native graph export (topology is implicit in the nodal Y-matrix); build the graph via DSS-Extensions Python APIs (BSD-3).
- **CIM (IEC 61970/61968) + CGMES (ENTSO-E):** the **authority’s declared grid model** in RDF/XML profile files (EQ/SSH/TP/SV/DL), “supports a node-breaker and a bus-branch model exchanges”; ENTSO-E publishes RDFS/SHACL artifacts under Apache-2.0. CIM→graph readers: **CIMantic Graphs (cim-graph)** and **CIMpy (SOGNO)**; or route CGMES → pandapower/PowSyBl → graph.

**Transit / metro maps:** **GTFS** is the transit authority’s declared network (TfL, BART, etc. publish it; >75% of scheduled-service agencies use it). Parsers: **gtfs-kit, partridge (fast, pandas-based), gtfs2geojson, gtfs-to-geojson, tidytransit (R)**; aggregators **Transitland / Mobility Database**. GTFS’s relational tables (routes/trips/stops/stop_times/shapes) ERD-map to nodes (stops) + edges (consecutive stop_times). **Embellishment flag:** automated octolinear/schematic metro-map layout is *research-grade* and *inferential* — if the agency publishes `shapes.txt` geometry, transcribe it; only synthesize a schematic layout as a marked-derived overlay.

**PCB / electronics:** **KiCad** netlist (S-expression) is the designer’s declared topology; parse with **kiutils (MIT)** or **kinparse**; **SKiDL (MIT)** can read/emit netlists and even DOT graphs.  Standards: SPICE netlists, EDIF, IPC-2581, ODB++ (the latter two are heavier industry formats). Nets→hyperedges, components/pins→nodes. **Flag:** a netlist is a hypergraph (a net connects many pins); transcribe it as the authority declared (net = super-node/sector) rather than inventing point-to-point edges.

**Documents:** **Pandoc AST (GPL — use as external CLI), unified/remark/markdown-it (MIT)** for Markdown ASTs, **Tree-sitter (MIT)**; **JATS/DocBook** for structured docs; headings→hierarchy (natural `child_manifest` recursion), links/citations→edges. Citation graphs from declared reference lists.

**General graph interchange the engine could ingest directly:** **GraphML** (XML, rich, schema-validated — the interoperability default, read by Gephi/yEd/NetworkX), **GEXF** (Gephi’s XML, read natively by sigma.js), **GML**, **DOT** (Graphviz), **JSON Graph Format** (jsongraph spec), **Cytoscape CX/JSON**, plus knowledge-graph routes: **RDF/SPARQL**, property graphs (**openCypher**, **Apache AGE** on Postgres, **DuckPGQ** SQL/PGQ on DuckDB).  **Recommendation:** define the cartridge as the canonical internal form, and provide thin importers from GraphML/GEXF/DOT/JSON-Graph so any conforming declared graph can be transcribed without bespoke code.

### Layer 5 — Provenance & honesty standards

- **Frictionless Data Package (`datapackage.json`, MIT-spec):** the closest existing standard to the cartridge contract — a required `resources` array (the manifest) listing data files with schemas and dataset-level metadata; the spec’s own contributors describe `resources` as “basically the manifest.” **Adopt it as the cartridge backbone**; manifest.json ≈ datapackage descriptor, nodes/edges/layers/sectors ≈ resources. Tabular Data Package + Table Schema give per-column typing and even cross-package foreign keys (a recursion hook). A `profile`/`$schema` field lets crawlers recognize the cartridge type.
- **JSON Schema:** author one schema per cartridge file (manifest/nodes/edges/layers/sectors/child_manifest) and validate every scanner’s output in CI. This is the mechanical guarantee that scanners “transcribe into a fixed contract without embellishing.”
- **W3C PROV / PROV-O (W3C Recommendation, 2013):** the domain-agnostic provenance ontology (Entity/Activity/Agent)  — attach to each cartridge a PROV record stating which authority declaration (entity) was read by which scanner (activity/agent), preserving per-scope findings and “data-derived truth.” **OpenLineage (LF AI & Data)** is the operational complement  for the build pipeline (Airflow/Spark/dbt integrations).
- **CloudEvents (CNCF, already named):** envelope for change/update events.
- **SDMX / DCAT:** statistical and dataset-catalog metadata standards — useful if cartridges are catalogued/discovered at scale (DCAT for the atlas-of-atlases index).
- **Content/identity:** **SWHID** (Software Hash Identifier) was “adopted as an international standard: ISO/IEC 18670:2025” on 23 April 2025 (V1.2) — use it for source artifacts; **Merkle/content-addressing** for cartridge integrity and dedup — content-hash each resource so the engine can verify a child cartridge matches its parent’s `child_manifest` pointer.

### Layer 6 — Precedents (“render any graph from a declarative source”)

- **Backstage software catalog (Apache-2.0):** entities + kinds + relations + spec form a declarative catalog graph rendered as an entity graph — a direct precedent for “authority declares structure, system renders it,” including a spec schema per entity.
- **Cytoscape.js compound nodes + expand-collapse (MIT):** the cleanest open-source precedent for recursive/drill-down “graph of graphs” (its expand-collapse extension is MIT, explicitly for “managing complexity of large compound graphs”).
- **Gephi / Gephi Lite:** render any GEXF/GraphML with layout — precedent for declarative-source rendering (Gephi Lite is browser-based on sigma.js).
- **Kepler.gl / deck.gl examples (MIT):** “map of any dataset” with declarative layer configs — precedent for layers.json.
- **DuckPGQ (MIT) / Apache AGE (Apache-2.0):** graph queries over columnar/relational stores; DuckPGQ implements SQL/PGQ (SQL:2023) on DuckDB. **Flag:** **Kùzu was archived 10 Oct 2025** (“This repository was archived by the owner… It is now read-only,” final release v0.11.3; the team cited “working on something new,” and Apple’s acquisition of Kùzu, Inc. later surfaced via the EU DMA list in Feb 2026). Community forks (“Ladybug”) are emerging, but for a durable, permissive path that aligns with your DuckDB/Parquet store, prefer **DuckPGQ** — accepting it is still a community extension under active development. 
- **Observable Plot / D3 (ISC/BSD):** declarative chart/graph specs — precedent for data-derived (never UI-authored) truth.
- **PowSyBl network-area-diagram / single-line-diagram (MPL-2.0):** precedent for an authority’s declared model rendered automatically as a diagram.

## Recommendations

**Stage 0 — Lock the contract (week 1–2).** Define the cartridge as a **Frictionless Data Package profile**: `manifest.json` (datapackage descriptor) + `nodes.json`/`edges.json` (GeoJSON resources) + `layers.json` + `sectors.json` + `child_manifest`. Write **JSON Schemas** for each and a validator that runs in CI. Add a **W3C PROV-O** stub to every cartridge and content-hash (Merkle) each resource. *Benchmark to proceed:* a hand-authored sample cartridge validates and renders.

**Stage 1 — Engine on federation scope first (weeks 2–6).** Build the scope-blind engine on **MapLibre GL JS** (flat plane, abstract coords in lon/lat slots), reading layers.json → MapLibre style layers and sectors.json → clusters. Wire **DuckDB-WASM + Parquet** as the proof store (Hive-partitioned by scope). Render the federation (top-level) cartridge — small enough for inline GeoJSON. *Benchmark:* federation atlas renders from static GitHub Pages with zero backend; clicking a node loads its `child_manifest`.

**Stage 2 — Repo-internals scanner to prove recursion (weeks 6–10).** Write the first scanner using **madge** or **dependency-cruiser** (+ **Syft** for the SBOM/cross-package layer) to transcribe a repo’s declared imports/deps into a cartridge, emitting a `child_manifest` per module so the engine recurses module → file → symbol. *Benchmark:* drilling from federation → repo → module works as Russian-doll recursion, each level a valid cartridge with its own provenance.

**Stage 3 — Scale tier + additional domain scanners (weeks 10+).** Add the **tippecanoe → PMTiles** build step so any scope exceeding ~50k features serves as vector tiles; add **deck.gl (OrthographicView)** or **cosmos.gl** as the WebGL overlay for 10^5–10^6-node interiors. Then add domain scanners in order of “most-declared topology”: **power (pandapower → NetworkX, or PowSyBl/CGMES)** → **transit (GTFS via gtfs-kit/partridge)** → **PCB (kiutils)** → **documents (unified/Pandoc)**. For each, transcribe declared geometry where present; mark any computed layout as derived in PROV.

**Decision thresholds that change the plan:**

- If a single scope routinely exceeds ~50k features → make PMTiles the default tier, not the exception.
- If interiors exceed ~10^6 nodes or you need runtime (not build-time) layout → promote **cosmos.gl** from overlay to primary renderer for that scope.
- If mobile-GPU testing shows WebGL instability (the cosmos.gl `OES_texture_float`/float-blend class of issue) → fall back to MapLibre vector tiles + simplified Canvas for those devices.
- If you need recursion-aware queries across scopes → adopt **DuckPGQ** SQL/PGQ over the Parquet store.

## Caveats

- **Embellishment is the central architectural risk.** The most powerful scanners (Tree-sitter/Joern CPG for code; octolinear metro-map layout for transit; AST call-graphs) *infer* topology the authority never declared. Enforce the rule that scanners transcribe only declared structure; any inferred edge or computed coordinate must be a separate, PROV-marked “derived” overlay, never presented as declared truth. The UI must never author truth — keep all truth data-derived in the cartridge/proof store.
- **Licensing traps to track:** **ELK = EPL-2.0** (weak copyleft; fine to use, obligations if modified); **OGDF, Pandoc, igraph, Gephi = GPL/AGPL family** (isolate as external build-time CLIs, never link/distribute); **GridCal = MPL-2.0 on master but some PyPI releases still LGPL** (pin versions and verify); **CodeQL / yFiles = proprietary** (exclude from the permissive core). The core runtime path (MapLibre BSD, PMTiles BSD, tippecanoe BSD, deck.gl MIT, cosmos.gl MIT, sigma MIT, DuckDB/Parquet/Arrow MIT/Apache, Cytoscape.js MIT) is cleanly MIT/BSD/Apache.
- **Maturity gaps:** **DuckPGQ** and **GeoParquet-in-DuckDB-WASM** are advancing fast but still maturing; **Kùzu is archived** (do not build on it); **cosmos.gl** is powerful but has documented mobile-WebGL fragility (no support on Android devices lacking `OES_texture_float`). Validate each on target devices before committing.
- **Scale-claim honesty:** vendor “1M node” claims (deck.gl, cosmos.gl) are real but device-dependent and assume simple per-item rendering; rich per-node styling, labels, and picking lower these ceilings substantially, and Chrome crashes most deck.gl layers between 10M and 100M items due to the 1 GB per-allocation cap. Budget for the SVG→Canvas→WebGL→vector-tile crossover to arrive earlier on mobile than the headline numbers suggest.
- **CIM/CGMES → GeoJSON has no turnkey open-source converter** as of mid-2026; expect custom scripting (via pandapower or PowSyBl intermediate models) to land grid topology on the map plane.
- I was unable to independently re-verify a few secondary scale figures (e.g., exact ngraph/G6 node ceilings and AntV G6’s WebGL-mode limits) within the research budget; treat those as directional rather than precise.
