Ten free-and-open external APIs to make the Spider the operative controller for GlobalGrid2050

TL;DR

- The ten highest-value, genuinely free-and-open external APIs to wire into the Spider, in ranked order, are: the NESO Data Portal (CKAN) including the Carbon Intensity API; Elexon Insights/BMRS (already a node, to be deepened rather than re-added); the six DNO open-data portals and their Embedded Capacity Registers, LTDS, flexibility and curtailment data (UKPN, NGED, SSEN, SP Energy Networks, Northern Powergrid, Electricity North West); the DESNZ Renewable Energy Planning Database (REPD); Ordnance Survey OS Data Hub OpenData APIs; Sheffield Solar PV_Live; the Octopus Energy REST API; National Highways data services; Network Rail open data feeds; and the British Geological Survey OGC API Features with its open geothermal point datasets. Every one is confirmed live in 2026 and carries an open licence (OGL, CC BY 4.0, or an ODbL-adjacent open licence) or a genuinely free tier with no commercial lock-in.
- Each enters the committed Parquet as a new external_source_or_service node carrying a licence field and an ingest/interoperate/align classification, with declared edges from the repos that consume it — chiefly data-gb-electricity, data-interconnectors, gb-electricity-ui and the globalgrid2050 monolith — directly enabling grid-connection viability screening, cable-route engineering, and market/price context, and generalising the delivered Braintree-style IEC 60287 cable-sizing capability nationally.
- Nothing enters the truth store until PO 1 is received and applied and Vikram accepts a quotation; this report stages that quotation only, honouring the one-in-one-out discipline.

Key Findings
The dominant finding is that almost the entire GB grid-connection and network-capacity picture is now available under open licences at zero cost, which means the Spider can be made genuinely operative for connection-viability screening and cable-route engineering without a single paid data contract. The named preference list maps almost perfectly onto live 2026 endpoints: NESO, all six DNOs, National Highways, Network Rail, the GLA, BGS and Octopus all publish open APIs. The friction points are not cost but three operational realities: authentication walls (several DNO portals gate the actual data table behind a free registration or API token; National Highways and Network Rail require registration), rate limits (NESO’s own guidance is a maximum of two requests per minute on the datastore and one request per second on the CKAN API; OS throttles live projects to 600 transactions per minute), and one genuine licence quirk (BGS’s national thermal-conductivity map is fee-licensed even though its borehole indexes and geothermal point data are open). Elexon BMRS is already a committed node, so the correct move is to deepen it, not consume a fresh one-in-one-out slot.

Details

Ranked recommendation 1 — NESO Data Portal and Carbon Intensity API (National Energy System Operator). This is the single most strategically central source and the natural spine of the federation, because it carries constraint costs and volumes, embedded wind and solar forecasts, the Transmission Entry Capacity / Interconnector / Embedded registers, the Future Energy Scenarios, and Strategic Spatial Energy Plan data, plus the official GB Carbon Intensity forecast. Data is delivered through a CKAN datastore API (curl-friendly resource downloads and datapackage_show endpoints) at api.neso.energy, and the Carbon Intensity API is a separate REST service documented at carbonintensity.org.uk. Licence: each dataset is licensed individually on the portal, and the Carbon Intensity API is explicitly CC BY 4.0. Cost: free. Access: REST/CKAN plus bulk CSV; the Carbon Intensity API needs no key; the NESO API guidance page states verbatim that for the “Datastore API: …we recommend limiting requests to a maximum of two requests per minute” and for the “CKAN API: It is recommended to limit requests to a maximum one request per second,” and My NESO accounts allow dataset subscriptions. Consuming repos: the globalgrid2050 monolith (already references NESO across 42 files), data-gb-electricity, and gb-electricity-ui (for the carbon feed, which the brief scopes separately). Declared edges: globalgrid2050 → NESO_data_portal; data-gb-electricity → NESO_data_portal; gb-electricity-ui → NESO_carbon_intensity_api. Classification: ingest (raw CSV/JSON pulled), then interoperate (aligned to the Parquet truth store), then align (mapped to atlas geometry and price estimators). Capability unlocked: connection-viability screening via constraint and embedded-generation context; market context via constraint costs; carbon context for the UI. Caveats: datasets are individually licensed, so each must carry its own licence field, and the two-per-minute datastore throttle means rollups should be cached rather than pulled live.

Ranked recommendation 2 — Elexon Insights / BMRS (Elexon Limited), already a node, to be deepened not re-added. The Insights Solution APIs serve FUELINST, FUELHH, prices and balancing data in CSV/XML/JSON from a production-grade REST service at data.elexon.co.uk; the Elexon API Developer Portal states verbatim that “All our APIs are public and no API key is required.” Licence: the BMRS Data Licence terms, free of charge to all users, attribution required, with scripting/screen-scraping outside the API prohibited. Note the legacy api.bmreports.com endpoints did require an Elexon portal scripting key, but the modern Insights REST API does not. Cost: free. Consuming repos: data-gb-electricity (partitioned zstd Parquet from Elexon — FUELINST, FUELHH, prices), data-interconnectors (named interconnector flows), and the globalgrid2050 monolith (Elexon/BMRS across 635 files). Declared edges already exist from these repos to the Elexon BMRS node; the outstanding work is the derived rollups for charts. Classification: ingest, then interoperate (derived rollups), then align. Capability: the core GB electricity time-series backbone and interconnector flow context. Caveat: it is already committed, so this is a deepening under the existing node, not a new one-in-one-out slot.

Ranked recommendation 3 — the six DNO open-data portals and their Embedded Capacity Registers, Long Term Development Statements, flexibility and curtailment data. This is where the preference list is densest and where connection-viability screening actually happens. UK Power Networks (ukpowernetworks.opendatasoft.com) runs on Opendatasoft with an Explore v2.1 REST API; its datasets sit under a Creative Commons CC BY 4.0-compatible licence (the portal uses the French Licence Ouverte, explicitly compatible with OGL, CC BY and ODC-BY)  with attribution required in a specified format (dataset name, source URL, date of last update); browsing is open but access to data tables requires a free registration . National Grid Electricity Distribution (connecteddata.nationalgrid.co.uk — covering the WPD-legacy East Midlands, West Midlands, South West and South Wales licence areas, now NGED) runs on CKAN under the NGED Open Data Licence, with curl-friendly resource downloads; from 1 June 2024 its API calls require an API token. SSEN (data.ssen.co.uk) runs a Datopian/CKAN distribution data portal plus the near-real-time NeRDA API. SP Energy Networks (spenergynetworks.opendatasoft.com) runs Opendatasoft with an Explore v2.1 API under its own Open Data Licence, publishing DFES, LTDS, GIS shapefiles, the Embedded Capacity Register (SPEN_006), Curtailment (SPEN_009) and Secondary Network Data. Northern Powergrid and Electricity North West both run Opendatasoft (free registration unlocks full data access and an increased API count). All six publish the ENA Open Networks-standard Embedded Capacity Register — generation and storage ≥50kW connected or accepted to connect — and Northern Powergrid additionally hosts a combined National Embedded Capacity Register merging all DNOs into one national view. Cost: free across all six. Access: REST (Opendatasoft Explore API or CKAN) plus bulk CSV/GeoJSON/shapefile, with free registration and per-portal API tokens for full data-table access. Consuming repos: the globalgrid2050 monolith foremost (atlas GeoJSON at 11kV–400kV, offtaker layers, cable and solar-BESS topology, price estimators), plus any connection-screening tooling. Declared edges: globalgrid2050 → each of six new nodes (UKPN_open_data, NGED_connected_data, SSEN_data_portal, SPEN_open_data, NPg_open_data, ENWL_open_data), or a single templated DNO node class carrying per-DNO licence fields. Classification: ingest (ECR/LTDS pulls), interoperate (aligned to atlas voltages and substation geometry), align (mapped to connection-viability screening). Capability unlocked: this is the core of connection viability — substation locations and headroom, embedded capacity registers, LTDS network capacity, flexibility tenders and curtailment tell you where a connection is viable and roughly what it costs. Caveats: registration and API-token walls on most portals; ECRs are updated monthly and can miss projects;  the exact point of connection is not published, so a DNO conversation is still required; and data-field definitions vary between DNOs , so the alignment work is real.

Ranked recommendation 4 — DESNZ Renewable Energy Planning Database (REPD), managed by Barbour ABI on behalf of DESNZ. The national register of every renewable electricity project above 150kW moving through planning, construction, operation and decommissioning, with installed capacity, technology type, CfD allocation round, planning status, grid coordinates and a full planning-milestone timeline. Licence: Open Government Licence v3.0. Cost: free. Access: quarterly bulk extract (Excel and CSV from gov.uk, plus KMZ/shapefile via third parties), with the latest being the Q1 2026 extract; there is an interactive map, but the canonical mechanism is bulk download rather than a REST API. Consuming repos: the globalgrid2050 monolith (REPD/DESNZ across 152 files). Declared edges: globalgrid2050 → DESNZ_REPD. Classification: ingest (quarterly CSV), interoperate (aligned to planning layers), align (mapped to the renewables pipeline view). Capability unlocked: planning data for renewables and pipeline context for connection screening. Caveats: bulk download rather than a live API, so a quarterly refresh cadence must be scheduled; and the 150kW threshold has applied only since 2021, so smaller pre-2021 projects are missing.

Ranked recommendation 5 — Ordnance Survey OS Data Hub OpenData APIs (Ordnance Survey). The survey-grade national geospatial baseline — OS Open Roads, OS OpenMap Local, Boundary-Line, Terrain 50 elevation, Code-Point Open postcodes, OS Open Names  — the definitive GB mapping that utilities and developers treat as the legal and operational baseline.  Licence: OS OpenData is free under the Open Government Licence; the premium tier (MasterMap building outlines and the like) is separate and out of scope for the free mandate, though a free £1,000/month premium allowance exists.  Cost: free for OpenData. Access: REST APIs (OS Features/NGD, OS Maps, OS Names, OS Downloads) returning GeoJSON, vector tiles and raster, plus bulk download; a free OS OpenData API key is required, and the OS Data Hub Plans page states verbatim that “All our API data (OS OpenData and Premium) are subject to a 600 transactions-per-minute throttle for your live projects… Projects in development mode… are further subjected to a 50 transactions-per-minute throttle.” Consuming repos: the globalgrid2050 monolith (atlas geometry, cable routing, transport layers). Declared edges: globalgrid2050 → OS_data_hub. Classification: ingest, interoperate (British National Grid alignment), align (atlas basemap and route geometry). Capability unlocked: cable-route engineering and land context — the road/rail/terrain baseline for trenching and corridor planning, and elevation for route optimisation. Caveats: an API key is required, and the OS Places (address) API contains Royal Mail IPR  and is not free, so stay on the OpenData products.

Ranked recommendation 6 — Sheffield Solar PV_Live (University of Sheffield, funded by NESO). Near-real-time and historical GB solar PV generation estimates, national and regional (by Grid Supply Point and by DNO licence area / PES region), at half-hourly resolution from 2010 to present. Licence: CC BY 4.0. Cost: free — the PV_Live-API project confirms NESO funds it “to ensure the data PV_Live provides can remain open and free.” Access: REST API (v4) at api.pvlive.uk  returning JSON, no API key, throttled per half hour with a 30-second response timeout (the v4 user guide notes “responses have a timeout (currently, 2025-01-05, 30s)”); a dedicated bulk endpoint exists for partner organisations such as NESO, and an official Python client (pvlive-api) is maintained. Consuming repos: gb-electricity-ui (the brief scopes solar PVLive separately) and the globalgrid2050 monolith (solar-BESS topology). Declared edges: gb-electricity-ui → sheffield_solar_pvlive; globalgrid2050 → sheffield_solar_pvlive. Classification: ingest (JSON pulls), interoperate (aligned to GSP/DNO geometry), align (mapped to solar-BESS estimators). Capability unlocked: solar generation context for the UI and for solar-BESS topology, generalising delivered PV work. Caveats: it excludes solar BMUs (metered separately by NESO via Elexon) to avoid double-counting;  and the base URL migrated to api.pvlive.uk in December 2024, with the v4 user guide stating the intent to “decommission the old api.solar.sheffield.ac.uk address/prefix in March 2025,” so use the current endpoint.

Ranked recommendation 7 — Octopus Energy REST API (Octopus Energy). Public half-hourly tariff pricing for Agile, Go and other products, tied to wholesale prices, plus product and tariff metadata. Licence: a genuinely free public API (developer.octopus.energy) with no commercial lock-in for the public pricing endpoints; consumption endpoints require the customer’s own API key, but the products/tariffs endpoints are unauthenticated. Cost: free. Access: REST at api.octopus.energy/v1, JSON, no key for product/tariff pricing, with pagination at 100 records. Consuming repos: the globalgrid2050 monolith (price estimators) and gb-electricity-ui (price context). Declared edges: globalgrid2050 → octopus_energy_api. Classification: ingest, interoperate (aligned to price estimators), align (mapped to market-context views). Capability unlocked: retail/market price context to complement Elexon wholesale and NESO constraint data — the market-context leg of the vision. Caveats: consumption data needs per-customer authentication (out of scope), and the pricing is Octopus-specific rather than a whole-market tariff feed.

Ranked recommendation 8 — National Highways data services (National Highways). Two complementary channels: the DATEX II traffic/roadworks/closure services via the developer portal (developer.data.nationalhighways.co.uk) and the National Highways Open Data GeoServices hub (opendata.nationalhighways.co.uk), which exposes the Strategic Road Network model, emergency areas and network features as feature layers. Licence: Open Government Licence (the data.gov.uk subset is explicitly OGL; the developer portal additionally requires acceptance of an Acceptable Use and Operational Policy). Cost: free. Access: for the developer portal, register and subscribe (subscriptions are set up individually with issued credentials); the DATEX II payload is XML (re-aligned to v3.4, with road-and-lane-closure data refreshed through the day and unplanned closures added from 2025); the Open Data hub offers CSV/KML/GeoJSON/GeoTIFF downloads and GeoServices/WMS/WFS API links. Consuming repos: the globalgrid2050 monolith (transport layers, cable routing). Declared edges: globalgrid2050 → national_highways_data. Classification: ingest (roadworks/SRN model), interoperate (aligned to route geometry), align (mapped to cable-route corridor screening). Capability unlocked: road context for cable routing — where the strategic road network runs, its closures and constraints — for open-air trough and trenching route planning. Caveats: the DATEX II live feeds require individual subscription and are XML (mapping effort); the OGL subset on data.gov.uk is lighter but less rich.

Ranked recommendation 9 — Network Rail open data feeds (Network Rail Infrastructure Limited). Real-time and static rail infrastructure and movement feeds (TRUST, TD, SCHEDULE, plus reference data) via the publicdatafeeds.networkrail.co.uk platform, with STOMP/ActiveMQ streaming and authenticated HTTP GET for static feeds. Licence: Network Rail open data terms and conditions (free). Cost: free. Access: register for an account, then connect via STOMP or HTTP GET; the Open Data Feeds page states verbatim that “Access to the feeds is provided on a first come, first served basis and is currently restricted to 1,000 users” (the service has been public since December 2012). Consuming repos: the globalgrid2050 monolith (transport layers, rail corridors for cable routing and HS2-adjacent context). Declared edges: globalgrid2050 → network_rail_feeds. Classification: ingest, interoperate (aligned to corridor geometry), align (mapped to route screening). Capability unlocked: rail corridor and land context for cable routing near railways. Caveats: registration is required and a 1,000-user cap applies; the feeds are movement/operational rather than geospatially asset-rich, so the routing value is corridor context rather than detailed asset data; and there is no dedicated HS2 open API, so HS2 corridor context must be inferred from OS, local-authority and other sources.

Ranked recommendation 10 — British Geological Survey OGC API Features and open geothermal datasets (BGS/UKRI). The OGC API Features service at ogcapi.bgs.ac.uk (live in 2026, labelled BETA, powered by pygeoapi) exposes the Single Onshore Borehole Index (SOBI,  over one million borehole records) and the AGS Borehole Records (third-party geotechnical logs) as GeoJSON with CQL query support, no API key and no documented rate limit. For cable thermal design, the open thermal data are point/lab datasets: the UK Digital Geothermal Catalogue Version 1 (subsurface temperature, rock thermal conductivity and heat flow — 11,821 points from 743 sites, OGL, CSV, DOI 10.5285/05569ed5-db0e-4587-807c-58e39ee240fa) and the “Thermal conductivity data on core samples 2025” dataset (OGL, Excel,  roughly nine boreholes). Licence: Open Government Licence for these, with the acknowledgement string confirmed on the UK Geothermal Platform as “Contains British Geological Survey materials © UKRI [year].” Cost: free for these. Access: OGC API Features REST/GeoJSON for the boreholes (no key), and bulk CSV/Excel download for the thermal point data. Consuming repos: the globalgrid2050 monolith (ground conditions for cable thermal design and trenching) and any tooling generalising a Braintree-style IEC 60287 thermal study nationally. Declared edges: globalgrid2050 → bgs_ogc_api. Classification: ingest (borehole/thermal pulls), interoperate (aligned to route geometry and thermal models), align (mapped to cable thermal design and trenching). Capability unlocked: ground conditions for IEC 60287 thermal design — borehole and ground data relevant to cable thermal resistivity and trenching, generalising delivered cable-sizing capability nationally. Caveats: the crucial licence quirk is that BGS’s national-coverage Thermal Properties Great Britain map (1:250,000 thermal conductivity, diffusivity, specific heat and density) is NOT open — it sits under the BGS Digital Data Licence  with a fee payable, and many UK Geothermal Platform layers are view-only under OGL, with the underlying data available only via a licence administration fee that the platform states verbatim is “currently £165+VAT” depending on usage type and area coverage (and BGS applied roughly a 3% rise to its direct licence fees from 1 April 2026). So BGS enters as an ingest-with-caveats node: the borehole indexes and geothermal point data are open, but there is no free spatially-continuous national soil thermal-conductivity grid, and a note should flag that the fee-licensed map may warrant a separate later quotation. The Met Office MIDAS Open soil temperature dataset (OGL, via CEDA) is a useful open complement for near-surface ground temperature.

Honourable mentions and why they rank below the top ten. The Greater London Authority London Datastore (OGL, DataPress/Datastore API) is strong but geographically limited to London and better filed as a local-authority node once the national pattern is proven. OpenStreetMap/Overpass and OpenInfraMap (ODbL) are already referenced across 79 files in the monolith and are effectively an existing pattern to formalise rather than a new discovery. Local authority open data (OGL, typically CKAN) is valuable for cable-route land context but is fragmented across hundreds of publishers and should enter as a templated node class rather than a single API. SSEN Transmission and the National Grid transmission portals reinforce the DNO pattern at transmission level and can be added under the same node template.

Ranking justification. The order follows two axes: fidelity to the named preference list, and leverage on the repo pattern. NESO and Elexon rank first and second because they are the market/system spine, Elexon is already a node, and NESO carries the connection registers and forecasts that almost everything else hangs off. The six DNO portals rank third as a bundle because they are the densest part of the preference list and the literal substance of connection-viability screening (headroom, ECR, LTDS, curtailment). REPD is fourth as the national renewables planning register already consumed across 152 files. OS is fifth as the geospatial baseline for all routing. PV_Live, Octopus, National Highways, Network Rail and BGS round out the ten because each unlocks a specific leg — solar, retail price, road, rail, ground conditions — that directly generalises the delivered Braintree-style engineering study, while each carries either an authentication wall or a partial licence quirk that places it below the clean, central sources.

Recommendations
Stage the quotation in tranches that respect one-in-one-out. First, deepen the existing Elexon node with the derived rollups (no new node needed) and add NESO as the first new external_source_or_service node, because it is central, cleanly CC BY 4.0 on the Carbon Intensity feed, and unblocks the most downstream work; the benchmark to proceed is a clean Parquet landing for data-gb-electricity. Second, once NESO is landed, add the DNO bundle as up to six nodes — or a single templated DNO node class carrying per-DNO licence fields — prioritising UKPN and NGED given the WPD-legacy footprint; the trigger is confirmation that the ECR/LTDS alignment schema holds across at least two DNOs. Third, add REPD and OS together as the planning-plus-basemap pair that unlocks connection screening overlaid on survey-grade geometry. Fourth, add PV_Live, Octopus, National Highways, Network Rail and BGS as capability-specific nodes as the corresponding repo needs mature (gb-electricity-ui for PV_Live and Octopus; the monolith’s transport and cable layers for Highways, Rail and BGS). The threshold that would change this staging: if any portal introduces a paid tier or revokes open access — watch the DNO API-token regimes and BGS licensing in particular — drop it and substitute the equivalent NESO or data.gov.uk mirror. Record every node’s licence field and ingest/interoperate/align classification at entry, and keep the provenance discriminator (declared versus derived, methodVersion federation_map_dna_v3_provenance_discriminator, edge key scanId+fromNode+toNode+edgeType+evidencePath) intact so each new declared edge is auditable.

Caveats
This is research to stage a quotation, not an instruction to apply anything: no new node enters the committed Parquet at data/federation_map/current until PO 1 is received and applied and a quotation is accepted by Vikram. Several sources sit behind free registration or API-token walls (most DNO portals, the National Highways developer portal, Network Rail), which is not a cost but is an operational dependency to record. Rate limits are real — NESO’s guidance is a maximum of two requests per minute on the datastore and one request per second on the CKAN API; OS throttles live projects to 600 transactions per minute (and development-mode projects to 50); PV_Live enforces a per-half-hour throttle with a 30-second timeout — all of which argue for cached derived rollups rather than live pulls. One genuine licence quirk stands out: BGS’s national thermal-conductivity map is fee-licensed (administration fee “currently £165+VAT,” raised roughly 3% from 1 April 2026) even though its borehole and geothermal point data are open, so cable thermal-design generalisation will rely on point data plus Met Office soil temperature until, or unless, a separate BGS map licence is quoted. REPD and DESNZ data are bulk-download quarterly rather than live APIs. Elexon is already committed, so its inclusion here is a deepening, not a new slot. Finally, all licences require attribution, so each node’s licence field must carry the exact acknowledgement string — for BGS, “Contains British Geological Survey materials © UKRI [year]”; for UKPN, the dataset-name-plus-URL-plus-date format; for the Carbon Intensity API, CC BY 4.0.
