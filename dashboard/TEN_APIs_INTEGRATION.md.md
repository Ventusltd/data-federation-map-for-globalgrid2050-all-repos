Operationalising the Ten Free-and-Open APIs into the GlobalGrid2050 Spider — an Auditor’s Deepening and Staged External-Source Quotation

TL;DR

- All ten APIs ranked in the committed BEST-API-for-VENTUS-SPIDER.md are live and open as of July 2026, and this report converts that ranking into executor-ready ingestion wiring — exact endpoints, resource IDs, authentication and rate limits — plus a federation-store landing pattern, while applying nothing before PO 1 closes.
- The ten sources should be staged into four sequential external-source-registry quotations honouring one-in-one-out: NESO first, the six DNOs as one templated node class second, REPD-plus-OS third, and PV_Live, Octopus, National Highways, Network Rail and BGS fourth, with Elexon deepened under its existing committed node rather than consuming a slot.
- The priority queue is fixed and must be worked in order: receive and apply PO 1 first, then documentation hygiene, then repo manifests and control-repo governance, then the external-source registry quotations in the proposed staging.

Key Findings

The central conclusion of the committed report holds and is reinforced by this deepening: cost is not the binding constraint, because every one of the ten sources is genuinely free and open as of July 2026. The binding constraints are three operational realities — authentication or registration walls on several portals, real rate limits that force cached derived rollups rather than live pulls, and a single licence quirk in the BGS estate. What this deepening adds is the concrete wiring that lets an executor purchase order implement ingestion without further research, and the exact federation-store landing pattern under the committed schema.

That schema is verified against the live DATA_CONTRACT.md in the control repository. The node grain is one row per discovered repository or external source node per scan, keyed on scanId plus nodeId, with required node fields scanId, nodeId, nodeKind, repoFullName, repoName, owner, repoType, status, visibility, defaultBranch, archived, htmlUrl, description, canonicalFilesPresent and generatedUTC.  The edge grain is one row per dependency edge per evidence source per scan, with required edge fields scanId, edgeId, fromNode, toNode, edgeType, cardinality, evidencePath, evidenceText and generatedUTC, and the natural key scanId plus fromNode plus toNode plus edgeType plus evidencePath,  alongside a scanId-plus-edgeId dedup key. Any external_source_or_service node and its declared edges must be expressed in exactly these fields, extended only with a licence field and an ingest-interoperate-align classification per the planned registry. This matters for the auditor seat because it means the external-source work does not invent a new object model; it reuses the committed one and adds two declared attributes.

Details

NESO Data Portal and Carbon Intensity API. The CKAN datastore is reached at api.neso.energy/api/3/action/datastore_search and datastore_search_sql, with resources fetched by resource_id, and datapackage_show used to enumerate a dataset’s resources. The verified first-pull resource identifiers relevant to GB grid-connection screening and market context are: the historic generation mix and carbon intensity dataset 88313ae5-94e4-4ddc-a790-593554d8c6b9 (file df_fuel_ckan.csv);  the national carbon intensity forecast resource 0e5fde43-2de7-4fb4-833d-c7bca3b658b0;  the regional carbon intensity forecast resource c16b0e19-c02a-44a8-ba05-4db2c0545a2a;  the 2-day-ahead demand forecast resource cda26f27-4bb6-4632-9fb5-2d029ca605e1;  the day-ahead national demand forecast resource aec5601a-7f3e-4c4c-bf56-d8e4184d3c5b;  the demand data update dataset carrying historic demand, interconnector, wind and solar outturn plus seven-day-ahead embedded wind and solar forecasts;  the 24-months-ahead constraint cost forecast resource 28b85d3f-a1cc-4bb9-80af-600f2cca266a;  the day-ahead constraint flows and limits dataset; and the constraint breakdown dataset splitting inertia, voltage and thermal costs and volumes, resource 6afe1c2b-6d70-4e76-8e74-0952b0a2beab.  The separate official Carbon Intensity REST service is at api.carbonintensity.org.uk (endpoints /intensity, /intensity/date and the /regional family), needs no key,  and is explicitly CC BY 4.0. On rate limits, the NESO API guidance page frames these as recommendations rather than hard caps: for the Datastore API it states “we recommend limiting requests to a maximum of two requests per minute,”  and for the CKAN API “it is recommended to limit requests to a maximum one request per second,”  with NESO reserving the right to block an IP address that overloads the server.  This argues decisively for cached derived rollups. This source lands as node NESO_data_portal, optionally with a companion NESO_carbon_intensity_api node for the cleanly CC-BY carbon feed, and declared edges from globalgrid2050, data-gb-electricity and gb-electricity-ui.

Elexon Insights/BMRS — deepen, do not re-add. The production REST service is at data.elexon.co.uk, and the Elexon developer portal states verbatim that all its APIs are public and no API key is required. The first-pull endpoints for GB screening and time-series backbone are the FUELINST instantaneous generation by fuel type, the half-hourly FUELHH, the Market Index Data MID price series (reported by APX and N2EX),  the settlement system prices by settlement date (DISEBSP) at /balancing/settlement/system-prices/{settlementDate}, and the interconnector outturn endpoints for data-interconnectors. The licence is the BMRS Data Licence, free of charge with attribution required  and scripting or screen-scraping outside the API prohibited; note that the legacy api.bmreports.com endpoints did require an Elexon scripting key but the modern Insights REST API does not. Because Elexon is already a committed node, this is a deepening — the derived rollups for charts — and it does not consume a one-in-one-out slot.

The six DNO open-data portals. UK Power Networks runs Opendatasoft Explore v2.1 at ukpowernetworks.opendatasoft.com, with the Embedded Capacity Register ≥1MW dataset identifier ukpn-embedded-capacity-register  and the 50kW-to-1MW register ukpn-embedded-capacity-register-1-under-1mw,  plus the related ukpn-large-embedded-power-stations (generation above 100MW connected or contracted)  and dfes-network-headroom-report (indicative unused demand and generation capacity at bulk supply and primary substations);  the catalog form is /api/v2/catalog/datasets/{id} and the query form /api/explore/v2.1/catalog/datasets/{id}/records. National Grid Electricity Distribution, covering the WPD-legacy East Midlands, West Midlands, South West and South Wales areas now branded NGED, runs CKAN at connecteddata.nationalgrid.co.uk with base call path /api/3/action/; since 1 June 2024 API calls require a token supplied in the Authorization HTTP header,  created from the user profile. Its LTDS Common Information Model dataset (slug ltds-common-information-model, dataset GUID 7367f8a5-7714-4647-84db-dc35119bea78) covers the extra-high-voltage systems at 132kV, 66kV and 33kV down to 11kV at primary substations, supporting Ofgem’s LTDS reform.  SSEN runs a Datopian/CKAN portal at data.ssen.co.uk with the embedded_capacity_register asset (Embedded Capacity Register 4.0, published June 2026)  plus the near-real-time NeRDA API at nerda.ssen.co.uk, which requires an account and exposes EHV, HV and LV power-flow data from over 5,000 secondary substations.  SP Energy Networks runs Opendatasoft at spenergynetworks.opendatasoft.com publishing the Embedded Capacity Register (dataset embedded-capacity-register, internal reference SPEN_006),  Curtailment (SPEN_009), Secondary Network Data (SPEN_015) and Technical Limits (SPEN_018), and LTDS covering its 132/33/11kV network. Northern Powergrid and Electricity North West both run Opendatasoft — Northern Powergrid at northernpowergrid.opendatasoft.com including the LTDS Appendix 6 fault-level dataset ltds-appendix-6  and hosting a combined National Embedded Capacity Register — with free registration unlocking full data access and increased API counts. All six publish the ENA Open Networks-standard ECR covering generation and storage of 50kW and above that is connected or accepted to connect.  The important engineering caveats are that ECRs are refreshed monthly and can miss projects, the exact point of connection is not published so a DNO conversation remains necessary, and field definitions vary between DNOs so the alignment work is real. These land preferably as a single templated DNO node class carrying per-DNO licence fields, or alternatively as six nodes (UKPN_open_data, NGED_connected_data, SSEN_data_portal, SPEN_open_data, NPg_open_data, ENWL_open_data), with declared edges chiefly from the globalgrid2050 monolith and any connection-screening tooling.

DESNZ Renewable Energy Planning Database. The canonical mechanism is the quarterly bulk extract on gov.uk rather than a live REST API; the current release is the Q1 2026 extract (data to end March 2026),  which replaced the January 2026 spreadsheets, last updated 6 May 2026, managed by Barbour ABI on behalf of DESNZ  under the Open Government Licence v3.0, delivered as CSV and XLSX with an interactive map at data.barbour-abi.com and third-party KMZ and shapefile mirrors. It records installed capacity, technology type, CfD allocation round, planning status, grid coordinates, planning authority and a full milestone timeline for every renewable electricity project above 150kW, a threshold that has applied only since 2021. It lands as node DESNZ_REPD with a scheduled quarterly refresh cadence.

Ordnance Survey OS Data Hub OpenData. The OpenData products are free under the Open Government Licence and are reached through REST APIs (OS Features and the newer NGD Features, OS Maps, OS Names, and OS Downloads) returning GeoJSON, vector tiles and raster, plus bulk download; a free OS OpenData API key is required, and the Downloads API can pull OpenData products without a key.   The routing-relevant products are OS Open Roads, OS OpenMap Local, Boundary-Line, OS Terrain 50 for elevation and route optimisation, Code-Point Open for postcode geometry, and OS Open Names. On throttling, the OS Data Hub Plans page states verbatim that “all our API data (OS OpenData and Premium) are subject to a 600 transactions-per-minute throttle for your live projects,”  while projects in development mode are further subject to a 50 transactions-per-minute throttle.   Stay on OpenData: the OS Places address API contains Royal Mail intellectual property and is fee-bearing. This lands as node OS_data_hub aligned to British National Grid (EPSG:27700).

Sheffield Solar PV_Live. The production REST v4 service is at api.pvlive.uk,  needs no key, is CC BY 4.0 and is funded by NESO to keep it open. The national endpoint is /pvlive/api/v4/gsp/0;  regional by Grid Supply Point is /pvlive/api/v4/gsp/{gsp_id}; regional by DNO licence area, also called the PES region, is /pvlive/api/v4/pes/{pes_id}; the enumerations are at /pvlive/api/v4/gsp_list and /pvlive/api/v4/pes_list; and an example query is api.pvlive.uk/pvlive/api/v4/gsp/0?start=2026-05-01T00:00:00&end=2026-05-05T00:00:00&data_format=csv.  The v4 user guide describes a per-half-hour request throttle and a 30-second response timeout,  notes that regionally-aggregated outturns currently (as of June 2026) use the 20260209 GSP boundary definitions hosted on the NESO portal,  and confirms that solar Balancing Mechanism Units are excluded to avoid double-counting with Elexon.  A maintained Python client, pvlive-api, exists. It lands as node sheffield_solar_pvlive with declared edges from gb-electricity-ui and the monolith.

Octopus Energy REST API. The base is api.octopus.energy/v1. The product list at /v1/products/ is unauthenticated  and paginated at 100 records;   product detail is /v1/products/{code}/; and half-hourly tariff pricing is /v1/products/{product}/electricity-tariffs/{tariff}/standard-unit-rates/ with period_from and period_to parameters in ISO 8601 format carrying a trailing Z for UTC.  The product and tariff endpoints require no key, while account and consumption endpoints require the customer’s own API key via HTTP Basic Auth (key as username, empty password) and are out of scope. Regional pricing varies by Grid Supply Point group, so the correct regional tariff code must be used. It lands as node octopus_energy_api for retail and market price context complementing Elexon wholesale and NESO constraint data.

National Highways. There are two complementary channels. The DATEX II developer portal at developer.data.nationalhighways.co.uk requires registration and per-service subscription with issued credentials and serves XML; its Road and Lane Closures service moved to v2.0 with a re-aligned DATEX II payload at greater cadence, and from late March 2025 it also provides closures from unplanned events, with v1.0 no longer updated as of May 2025.  The Open Data GeoServices hub at opendata.nationalhighways.co.uk exposes the Network Model — a daily-updated digital representation of the Strategic Road Network as links and nodes, derived from Ordnance Survey Highways data and enriched with internal datasets — plus emergency areas and diversion routes, downloadable as CSV, KML, GeoJSON and GeoTIFF with WMS, WFS and GeoServices API links, under the Open Government Licence for the data.gov.uk subset. It lands as node national_highways_data for open-air trough and trenching corridor screening.

Network Rail open data feeds. Registration is at publicdatafeeds.networkrail.co.uk; real-time feeds (TRUST, TD, VSTP, RTPPM and reference data) are consumed via STOMP, OpenWire or AMQP on TCP port 61618, and static feeds via authenticated HTTP GET — HTTP polling of the streaming feeds does not work, as the service is an event-based message queue, not a REST API. The Network Rail open-data-feeds page states verbatim that “access to the feeds is provided on a first come, first served basis and is currently restricted to 1,000 users,”  and the service has been public since December 2012. The feeds are movement and operational rather than geospatially asset-rich, so the value here is rail-corridor context rather than detailed asset data, and there is no dedicated HS2 open API, so HS2 corridor context must be inferred from OS, local-authority and other sources. It lands as node network_rail_feeds.

British Geological Survey OGC API Features and open geothermal data. The OGC API Features service (labelled BETA, powered by pygeoapi) is at ogcapi.bgs.ac.uk, returns GeoJSON with CQL filter support, needs no key and publishes no documented rate limit. The collections relevant to cable thermal design and route ground conditions are onshoreboreholeindex (the Single Onshore Borehole Index, over one million borehole, shaft and well records back to at least 1790), agsboreholeindex (third-party AGS-format geotechnical logs delivered as received), and bgsgeology625ksuperficial (superficial deposits). A representative query is ogcapi.bgs.ac.uk/collections/onshoreboreholeindex/items?f=json&limit=100 with CQL filters, and the service is compatible with QGIS and ArcPro for a live link. The open thermal point data are the UK Digital Geothermal Catalogue Version 1 and the Thermal conductivity data on core samples 2025 dataset. Per BGS, the geothermal catalogue “contains 11 821 data points derived from 743 sites, comprising 77 per cent of the legacy geothermal catalogue dataset held by BGS,”  released under the Open Government Licence v3.0 with the acknowledgement “Contains British Geological Survey materials © UKRI 2024,”  carrying DOI 10.5285/05569ed5-db0e-4587-807c-58e39ee240fa.  The crucial licence quirk is that the spatially-continuous Thermal Properties Great Britain version 1 map — a 1:250,000-scale vector polygon coverage attributed with thermal conductivity, thermal diffusivity, specific heat capacity and density, DOI 10.5285/def961e0-3432-4af8-a09a-a489c845af54 — is NOT open; it is available only under the BGS data licence for a fee. So BGS enters as an ingest-with-caveats node bgs_ogc_api: the borehole indexes and geothermal point data are open, but there is no free spatially-continuous national soil thermal-conductivity grid, and the fee-licensed map should be flagged for a possible separate later quotation. The Met Office MIDAS Open soil-temperature dataset (OGL, via CEDA) is a useful open complement for near-surface ground temperature.

How each source lands in the truth store. Each source becomes an external_source_or_service node carrying nodeId, nodeKind set to external_source_or_service, the standard scan fields defined above, a new licence field holding the exact acknowledgement string, and the ingest-interoperate-align classification. Declared edges run from the consuming repos — data-gb-electricity, data-interconnectors, gb-electricity-ui, the globalgrid2050 monolith as archive, and future data repos — to the source node, using the committed edge fields scanId, edgeId, fromNode, toNode, edgeType, cardinality, evidencePath and evidenceText, with provenance recorded as declared. Because sibling repos currently lack federation.yaml manifests, the scanner today would have to infer these edges from README and DATA_SOURCES mentions and known API references; the correct pattern is for each consuming repo to declare which external sources it consumes in its own federation.yaml manifest,  so scanner deduction becomes declared local fact rather than inference. This is the mechanism that connects the API work to the coherence findings.

The staged quotation structure. Split the ten into four sequential external-source-registry quotations, taken one at a time under one-in-one-out. Quotation A adds NESO as the first new node, because it is the central spine, cleanly CC BY 4.0 on the carbon feed, and unblocks the most downstream work; Elexon is deepened under its existing node in parallel and does not consume a slot. Quotation B adds the six DNO portals as one templated DNO node class carrying per-DNO licence fields, prioritising UKPN and NGED given the WPD-legacy footprint. Quotation C adds REPD and OS together as the planning-plus-basemap pair that lets connection screening be overlaid on survey-grade geometry. Quotation D adds PV_Live, Octopus, National Highways, Network Rail and BGS as capability-specific nodes as the corresponding repo needs mature. The named acceptance checks for an external-source-registry PO, patterned on the style of the existing twelve PO 1 checks, should include source_node_exists_with_licence_field, edge_provenance_declared_only, no_data_fetched_without_accepted_ingest_po, registry_deterministic_from_clean_clone, licence_acknowledgement_string_present, ingest_interoperate_align_classification_present, and consuming_repo_manifest_declares_source. The Spider UI should render external sources as nodes bearing a licence badge derived only from the committed licence field, never hand-authored, honouring the standing doctrine that the internal Jean Luc dashboard must read from verified Parquet-derived JSON only  and must not infer truth or hand-author status colours.

Coherence findings from the architectural study, and how the API work interacts with them. The monolith README carries stale pre-federation doctrine, and the control repo README itself signals that historical homepage doctrine is context only and that the ledger is the permanent, self-sufficient source of truth. Sibling repos lack federation.yaml manifests, and the control repo lacks LICENSE, GOVERNANCE.md, SECURITY.md and CODEOWNERS. The node taxonomy should distinguish core-grid-product, platform, reference-fork, training and research, and the external-source work adds the external_source_or_service kind on top. The interaction is direct: manifests are the mechanism that turns each external-source edge from scanner inference into a declared local fact, so the manifest and governance hygiene is a genuine prerequisite for the registry work being auditable rather than deduced. Adding the external-source nodes before the manifests exist would force the scanner to infer consumption, which is exactly the provenance weakness the declared-versus-derived discipline exists to prevent.

The engineering mission. These ten sources generalise the Ventus-funded Braintree EV Forecourt cable-sizing methodology — IEC 60287 thermal current ratings, IEC 60909 fault-level analysis, ERA 69-30 trough and buried-cable rating analysis, 33kV practice, and Ventus 110kV capability portable to GB 132kV — into a national grid-connection screening capability. BGS supplies the ground-conditions context that drives thermal resistivity: ERA 69-30 assumes a soil thermal resistivity of 1.2 K·m/W, corresponding to typical UK soil in Annex A of BS IEC 60287-3-1, and drying of backfill toward 2.5 K·m/W can cut ampacity substantially, so borehole logs and superficial-deposit mapping materially change a rating and are the right open inputs for the thermal-resistivity context (with the fee-licensed thermal-conductivity map noted as an optional later purchase). The DNO LTDS and embedded capacity registers supply the network fault-level and headroom context — NGED’s LTDS CIM at 132/66/33/11kV, Northern Powergrid’s LTDS Appendix 6 fault levels, and all six ECRs with substation locations and indicative headroom — which is the literal substance of connection viability. NESO and Elexon supply the market and constraint context, Octopus adds retail price context, and OS, National Highways and Network Rail supply the route geometry and corridor context for cable-route engineering. Throughout, the interface must display the chartered-engineer boundary between screening and certified design: these open sources support indicative national screening only, and any output must be clearly marked as not a certified engineering design.

Caveats

This is research that stages a quotation; it applies nothing, and no new node enters the committed Parquet until PO 1 is received and applied and a quotation is accepted. Several sources sit behind free registration or API-token walls — most DNO portals, the National Highways developer portal, and Network Rail — which is an operational dependency to record rather than a cost. Rate limits are real and argue for cached derived rollups rather than live pulls: NESO recommends a maximum of two requests per minute on the Datastore and one per second on the CKAN API and reserves the right to block overloading IPs;  OS throttles live projects to 600 transactions per minute  and development-mode projects to 50;  and PV_Live enforces a per-half-hour throttle with a 30-second timeout.  The BGS national thermal-conductivity map is fee-licensed even though the borehole indexes and geothermal point data are open. REPD is a quarterly bulk download rather than a live API, so a refresh cadence must be scheduled. Elexon is already committed, so its inclusion here is a deepening, not a new one-in-one-out slot. All licences require attribution, so each node’s licence field must carry the exact acknowledgement string — for BGS “Contains British Geological Survey materials © UKRI [year],”  for UKPN the dataset-name-plus-URL-plus-date format, and CC BY 4.0 for the Carbon Intensity API and PV_Live.

Finally, a clean separation of declared from verified for the auditor record. The following are verified against live sources as of July 2026: the ten APIs’ endpoints, resource IDs, authentication, rate limits and licences as set out above; and the committed node and edge field lists, grain and keys, which are verified from the live DATA_CONTRACT.md.
